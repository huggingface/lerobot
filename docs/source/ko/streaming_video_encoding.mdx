# 스트리밍 비디오 인코딩 가이드

## 1. 개요

스트리밍 비디오 인코딩(Streaming Video Encoding)은 비디오 데이터셋 녹화 시 기존에 거치던 PNG 왕복(Round-trip) 과정을 제거합니다. 기존 방식은 다음과 같았습니다.

1. 프레임 캡처 -> 디스크에 PNG 기록 -> (에피소드 종료 시) PNG 읽기 -> MP4로 인코딩 -> PNG 삭제

이제 프레임을 캡처하는 동시에 실시간으로 인코딩할 수 있습니다.

1. 프레임 캡처 -> 인코더 스레드(Encoder Thread)로 큐(Queue) 전송 -> MP4로 바로 인코딩

이로 인해 `save_episode()`는 거의 즉시 완료되며(에피소드가 끝나는 시점에 비디오는 이미 인코딩이 완료된 상태입니다), 특히 다중 카메라로 긴 에피소드를 녹화할 때 에피소드 사이에 발생하던 블로킹 대기(Blocking Wait)가 사라집니다.

## 2. 튜닝 파라미터

| 파라미터                | CLI 플래그                        | 타입          | 기본값        | 설명                                                              |
| ----------------------- | --------------------------------- | ------------- | ------------- | ----------------------------------------------------------------- |
| `streaming_encoding`    | `--dataset.streaming_encoding`    | `bool`        | `True`        | 캡처 중 실시간 인코딩 활성화                                      |
| `vcodec`                | `--dataset.vcodec`                | `str`         | `"libsvtav1"` | 비디오 코덱. `"auto"`는 최적의 HW 인코더를 자동 감지              |
| `encoder_threads`       | `--dataset.encoder_threads`       | `int \| None` | `None` (auto) | 인코더 인스턴스당 스레드 수. `None`이면 vcodec이 결정하도록 위임  |
| `encoder_queue_maxsize` | `--dataset.encoder_queue_maxsize` | `int`         | `60`          | 카메라당 최대 버퍼 프레임 수 (30fps에서 약 2초). RAM을 소비함     |

## 3. 성능 고려 사항

스트리밍 인코딩은 캡처 루프(Capture Loop) 이후가 아니라 **캡처 루프 중에** CPU가 비디오를 인코딩한다는 의미입니다. 이로 인해 다음 항목들 간에 공유해야 하는 CPU 예산(Budget)이 생깁니다.

- **제어 루프(Control Loop)** (카메라 읽기, 로봇 제어, 비디오 외 데이터 기록)
- **인코더 스레드** (카메라당 하나의 풀(Pool))
- **Rerun 시각화** (활성화된 경우)
- **OS 및 기타 프로세스**

### 해상도와 카메라 수의 영향

| 설정                      | 처리량 (px/sec) | CPU 인코딩 부하 | 비고                                |
| ------------------------- | --------------- | --------------- | ----------------------------------- |
| 2camsx 640x480x3 @30fps   | 55M             | 낮음            | 대부분의 시스템에서 동작            |
| 2camsx 1280x720x3 @30fps  | 165M            | 중간            | 최신 시스템에서 무난                |
| 2camsx 1920x1080x3 @30fps | 373M            | 높음            | 강력한 하이엔드 CPU 필요            |

### `encoder_threads` 튜닝

이 파라미터는 각 인코더 인스턴스가 내부적으로 사용하는 스레드 수를 제어합니다.

- **높은 값** (예: 4-5): 인코딩 속도는 빨라지지만, 카메라당 더 많은 CPU 코어를 사용합니다. 코어 수가 많은 하이엔드 시스템에 적합합니다.
- **낮은 값** (예: 1-2): 카메라당 CPU 사용량이 적어 캡처와 시각화를 위한 코어를 확보할 수 있습니다. 저해상도 이미지와 성능 좋은 CPU에 적합합니다.
- **`None` (기본값)**: 코덱이 자체적으로 결정하도록 합니다. 관련 정보는 코덱 로그에서 확인할 수 있습니다.

### 백프레셔(Backpressure)와 프레임 드롭(Frame Dropping)

각 카메라는 제한된 크기의 큐(`encoder_queue_maxsize`, 기본값 60 프레임)를 가집니다. 인코더가 따라가지 못할 경우 다음과 같이 동작합니다.

1. 큐가 가득 찹니다 (RAM 소비).
2. 새 프레임은 **드롭(Drop)**됩니다 (블로킹되지 않음) — 캡처 루프는 중단 없이 계속 진행됩니다.
3. 다음과 같은 경고가 로그에 기록됩니다: `"Encoder queue full for {camera}, dropped N frame(s)"`
4. 에피소드 종료 시, 카메라별 총 드롭된 프레임 수가 보고됩니다.

### 인코더가 뒤처질 때의 증상

- **시스템이 끊기거나 멈춤**: 모든 CPU가 100% 사용 중
- **드롭된 프레임 경고**가 로그에 나타나거나, 녹화된 데이터셋의 프레임 수/FPS가 예상보다 낮음
- **로봇 움직임이 끊김**: CPU가 심각하게 과부하 상태일 경우, 캡처 루프까지 영향을 받을 수 있음
- **rerun 지연 누적**: 시각화가 실시간을 따라가지 못함

## 4. 하드웨어 가속 인코딩

### 사용 시점

다음과 같은 경우 HW 인코딩을 사용하시기 바랍니다.

- CPU가 병목(Bottleneck) 지점인 경우 (프레임 드롭, 로봇 움직임 끊김, rerun 지연 발생)
- 호환 가능한 하드웨어(GPU 또는 전용 인코더)를 보유한 경우
- 높은 처리량(고해상도 또는 다중 카메라)으로 녹화하는 경우

### 코덱 선택

| 코덱                  | CPU 사용량 | 파일 크기      | 품질  | 비고                                                              |
| --------------------- | ---------- | -------------- | ----- | ----------------------------------------------------------------- |
| `libsvtav1` (기본값)  | 높음       | 가장 작음      | 최상  | 기본값. 최고의 압축률이지만 CPU 사용량이 가장 많음                |
| `h264`                | 중간       | 약 30-50% 큼   | 양호  | 소프트웨어 H.264. CPU 사용량 낮음                                 |
| HW 인코더             | 매우 낮음  | 가장 큼        | 양호  | 전용 하드웨어로 오프로드. CPU 제약이 큰 시스템에 가장 적합        |

### 사용 가능한 HW 인코더

| 인코더              | 플랫폼        | 하드웨어                                                                                              | CLI 값                               |
| ------------------- | ------------- | ----------------------------------------------------------------------------------------------------- | ------------------------------------ |
| `h264_videotoolbox` | macOS         | Apple Silicon / Intel                                                                                 | `--dataset.vcodec=h264_videotoolbox` |
| `hevc_videotoolbox` | macOS         | Apple Silicon / Intel                                                                                 | `--dataset.vcodec=hevc_videotoolbox` |
| `h264_nvenc`        | Linux/Windows | NVIDIA GPU                                                                                            | `--dataset.vcodec=h264_nvenc`        |
| `hevc_nvenc`        | Linux/Windows | NVIDIA GPU                                                                                            | `--dataset.vcodec=hevc_nvenc`        |
| `h264_vaapi`        | Linux         | Intel/AMD GPU                                                                                         | `--dataset.vcodec=h264_vaapi`        |
| `h264_qsv`          | Linux/Windows | Intel Quick Sync                                                                                      | `--dataset.vcodec=h264_qsv`          |
| `auto`              | 모든 플랫폼   | 시스템에서 사용 가능한 HW 인코더를 탐색. HW 인코더가 없으면 `libsvtav1`로 폴백(Fallback)              | `--dataset.vcodec=auto`              |

> [!NOTE]
> HW 가속 인코더를 사용하기 위해서는 GPU 드라이버 업그레이드가 필요할 수 있습니다.

> [!NOTE]
> `libsvtav1`이 기본값인 이유는 최상의 학습 성능을 제공하기 때문입니다. 다른 vcodec은 CPU 사용량을 줄이고 더 빠를 수 있지만, 일반적으로 더 큰 파일을 생성하며 학습 시간에 영향을 미칠 수 있습니다.

## 5. 문제 해결

| 증상                                                                       | 가능성 있는 원인                                  | 해결 방법                                                                                                                                                                                                                                                                          |
| -------------------------------------------------------------------------- | ------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 시스템 멈춤, 로봇 움직임 끊김, 또는 Rerun 시각화 지연                      | CPU 부족 (100% 부하 사용)                         | 다른 앱을 닫고, 인코딩 처리량을 줄이며, `encoder_threads`를 낮추거나, `h264`를 사용하거나, `display_data=False`를 사용합니다. CPU가 계속 100% 상태라면 시스템 사양이 부족할 수 있으므로 `--dataset.streaming_encoding=false` 또는 HW 인코딩(`--dataset.vcodec=auto`)을 고려하시기 바랍니다. |
| "Encoder queue full" 경고 또는 데이터셋에서 프레임 드롭 발생               | 인코더가 따라가지 못함 (큐 오버플로우)            | CPU가 100%가 아닐 경우: `encoder_threads`를 늘리거나, `encoder_queue_maxsize`를 늘리거나, HW 인코딩(`--dataset.vcodec=auto`)을 사용하시기 바랍니다.                                                                                                                                  |
| 높은 RAM 사용량                                                            | 인코딩보다 큐가 더 빨리 채워짐                    | `encoder_threads`가 너무 낮거나 CPU가 부족한 상태입니다. `encoder_queue_maxsize`를 줄이거나 HW 인코딩을 사용하시기 바랍니다.                                                                                                                                                        |
| 큰 비디오 파일                                                             | HW 인코더 또는 H.264 사용 중                      | 예상되는 트레이드오프(Trade-off)입니다. CPU 여유가 있다면 `libsvtav1`로 전환하시기 바랍니다.                                                                                                                                                                                       |
| `save_episode()`가 여전히 느림                                             | `streaming_encoding`이 `False`                    | `--dataset.streaming_encoding=true`로 설정하시기 바랍니다.                                                                                                                                                                                                                         |
| 인코더 스레드 크래시(Crash)                                                | 코덱이 사용 불가능하거나 설정이 잘못됨            | `vcodec`이 설치되어 있는지 확인하고, `--dataset.vcodec=auto`를 시도하시기 바랍니다.                                                                                                                                                                                                |
| 녹화된 데이터셋에 프레임 누락                                              | CPU/GPU 부족 또는 일시적 부하 급증                | 약 5%의 프레임이 누락된다면 시스템이 과부하 상태일 가능성이 높으므로 위의 권장 사항을 따르시기 바랍니다. 더 적은 수(~2%)가 누락된다면 일시적 부하 급증(주로 시작 시점)에 의한 것일 수 있으며, 이는 예상 범위 내로 간주할 수 있습니다.                                              |

## 6. 권장 설정

이 추정치는 보수적으로 설정된 값이므로, 본인의 시스템에서 테스트해 보시기 바랍니다. 낮은 부하에서 시작하여 점진적으로 늘려가는 것을 권장합니다.

### 하이엔드 시스템: 최신 12코어 이상 (24스레드 이상)

CPU에서 약 250-500M px/sec의 처리량을 무난하게 처리할 수 있습니다. 더 나은 결과를 원한다면 가능한 경우 HW 인코딩을 시도해 보시기 바랍니다.

```bash
# 3camsx 1280x720x3 @30fps: 기본 설정으로도 잘 동작합니다. 선택적으로 인코더 병렬화 수준을 높일 수 있습니다.
# 2camsx 1920x1080x3 @30fps: 기본 설정으로도 잘 동작합니다. 선택적으로 인코더 병렬화 수준을 높일 수 있습니다.
lerobot-record --dataset.encoder_threads=5 ...

# 3camsx 1920x1080x3 @30fps: 약간의 튜닝이 필요할 수 있습니다.
```

### 미드레인지 시스템: 최신 8코어 이상 (16스레드 이상) 또는 Apple Silicon

CPU에서 약 80-300M px/sec의 처리량이 가능합니다.

```bash
# 3camsx 640x480x3 @30fps: 기본 설정으로도 잘 동작합니다. 선택적으로 인코더 병렬화 수준을 낮출 수 있습니다.
# 2camsx 1280x720x3 @30fps: 기본 설정으로도 잘 동작합니다. 선택적으로 인코더 병렬화 수준을 낮출 수 있습니다.
lerobot-record --dataset.encoder_threads=2 ...

# 2camsx 1920x1080x3 @30fps: 약간의 튜닝이 필요할 수 있습니다.
```

### 리소스가 제한된 시스템: 최신 4코어 이상 (8스레드 이상) 또는 Raspberry Pi 5

매우 제약된 시스템에서는 스트리밍 인코딩이 캡처 루프와 지나치게 경쟁할 수 있습니다. 이를 비활성화하면 에피소드 사이에 인코딩이 수행되는 PNG 기반 방식으로 폴백됩니다(블로킹이 발생하지만 캡처에 간섭하지는 않습니다). 또는 캡처와 인코딩 부하를 모두 줄이기 위해 낮은 처리량으로 녹화할 수도 있습니다. 코덱을 `h264`로 변경하고 배치 인코딩(Batch Encoding)을 사용하는 것도 고려해 보시기 바랍니다.

```bash
# 2camsx 640x480x3 @30fps: 약간의 튜닝이 필요합니다.

# H.264 사용, 스트리밍 비활성화, 배치 인코딩 고려
lerobot-record --dataset.vcodec=h264 --dataset.streaming_encoding=false ...
```

## 7. 마무리

성능은 결국 당신의 정확한 설정에 따라 달라집니다 — 초당 프레임 수(FPS), 해상도, CPU 코어와 부하, 사용 가능한 메모리, 에피소드 길이, 그리고 선택한 인코더에 따라 결정됩니다. 항상 목표 워크로드(Workload)로 테스트하고, CPU 및 시스템 성능을 염두에 두며, `encoder_threads`, `encoder_queue_maxsize`, `vcodec`을 합리적으로 튜닝하시기 바랍니다. 한편, 많은 응용 분야에서 일반적으로 사용되는 실용적인 구성은 3대의 카메라를 640×480x3 @30fps로 운용하는 것이며, 이는 최신 시스템의 기본 스트리밍 비디오 인코딩 설정에서 보통 무리 없이 동작합니다. 녹화된 데이터셋이 정상인지 항상 검증하시기 바라며, 이는 비디오 길이와 CLI 에피소드 길이를 비교하고 행(Row) 개수가 FPS × CLI 길이와 같은지 확인하는 방식으로 수행합니다.
