# HIL-SERL 실제 로봇 학습 워크플로우 가이드

이 튜토리얼에서는 LeRobot으로 Human-in-the-Loop Sample-Efficient Reinforcement Learning(HIL-SERL) 전체 워크플로우를 따라갑니다. 몇 시간 안에 실제 로봇에서 RL로 정책을 학습시키는 방법을 익힐 수 있습니다.

HIL-SERL은 사람 시연, 온라인 학습, 사람 개입을 결합한 샘플 효율형 강화학습 알고리즘입니다. 소량의 사람 시연으로 시작해 보상 분류기를 학습하고, 이후 액터-러너(actor-learner) 구조에서 정책을 실행하는 동안 사람이 필요할 때 개입해 탐색을 유도하고 위험한 동작을 교정합니다. 이 튜토리얼에서는 게임패드를 사용해 개입을 제공하고 학습 중 로봇을 제어합니다.

핵심은 다음 세 가지입니다.

1. **오프라인 시연 + 보상 분류기:** 소량의 텔레오퍼레이션 시연 에피소드와 비전 기반 성공 탐지기로 정책 학습의 시작점을 안정적으로 만듭니다.

2. **실로봇 액터/러너 루프 + 사람 개입:** 분산 Soft Actor Critic(SAC) 러너가 정책을 업데이트하고 액터는 실제 로봇에서 탐색합니다. 사람은 언제든 개입해 위험하거나 비효율적인 동작을 수정할 수 있습니다.

3. **안전/효율 도구:** 관절/엔드이펙터 경계, ROI 크롭 전처리, WandB 모니터링으로 데이터 품질과 하드웨어 안전을 함께 확보합니다.

이 요소들을 조합하면 HIL-SERL은 모방학습 기반 기준선 대비 거의 완벽에 가까운 작업 성공률과 더 빠른 학습 사이클을 달성할 수 있습니다.

## 무엇이 필요한가요?

- 로봇 제어용 게임패드(권장) 또는 키보드
- Nvidia GPU
- 실제 로봇(팔로워/리더 암 구성, 키보드/게임패드만 쓸 경우 선택)
- 로봇용 URDF 파일(참고: lerobot/model/kinematics.py)

## 어떤 작업을 학습할 수 있나요?

HIL-SERL은 다양한 조작 작업에 적용할 수 있습니다. 시작은 아래처럼 권장합니다.

- 먼저 간단한 작업으로 시스템 동작을 이해해주세요.
  - 큐브를 목표 영역으로 밀기
  - 그리퍼로 큐브를 집어 들어 올리기
- 너무 긴 horizon 작업은 피해주세요. 5~10초 내 완료 가능한 작업에 집중하는 것이 좋습니다.
- 시스템에 익숙해지면 더 복잡한 작업과 긴 horizon으로 확장해주세요.
  - 큐브 pick and place
  - 양팔로 물체를 집는 bimanual 작업
  - 한 팔에서 다른 팔로 물체를 넘기는 hand-over 작업
  - 자유롭게 확장해 주세요.

## 실제 로봇 학습 워크플로우

### 설정 구조 이해하기

학습은 HILSerl 환경 설정에서 시작합니다. 핵심 설정 클래스는 lerobot/rl/gym_manipulator.py의 GymManipulatorConfig이며, 내부에 HILSerlRobotEnvConfig와 DatasetConfig가 중첩되어 있습니다. 설정은 역할별로 분리된 중첩 구조를 따릅니다.

<!-- prettier-ignore-start -->
```python
class GymManipulatorConfig:
    env: HILSerlRobotEnvConfig    # 환경 설정(중첩)
    dataset: DatasetConfig    # 데이터셋 기록/재생 설정(중첩)
    mode: str | None = None    # "record", "replay", 또는 None(학습용)
    device: str = "cpu"    # 연산 장치

class HILSerlRobotEnvConfig(EnvConfig):
    robot: RobotConfig | None = None    # 메인 로봇 에이전트(`lerobot/robots`에 정의)
    teleop: TeleoperatorConfig | None = None    # 텔레오퍼레이터 에이전트(예: 게임패드, 리더 암)
    processor: HILSerlProcessorConfig    # 처리 파이프라인 설정(중첩)
    name: str = "real_robot"    # 환경 이름
    task: str | None = None    # 작업 식별자
    fps: int = 10    # 제어 주파수

# 중첩된 processor 설정
class HILSerlProcessorConfig:
    control_mode: str = "gamepad"    # 제어 모드
    observation: ObservationConfig | None = None    # 관측 처리 설정
    image_preprocessing: ImagePreprocessingConfig | None = None    # 이미지 크롭/리사이즈 설정
    gripper: GripperConfig | None = None    # 그리퍼 제어 및 패널티 설정
    reset: ResetConfig | None = None    # 환경 리셋 및 타이밍 설정
    inverse_kinematics: InverseKinematicsConfig | None = None    # IK 처리 설정
    reward_classifier: RewardClassifierConfig | None = None    # 보상 분류기 설정
    max_gripper_pos: float | None = 100.0    # 최대 그리퍼 위치

# 하위 설정 클래스
class ObservationConfig:
    add_joint_velocity_to_observation: bool = False    # 상태에 관절 속도 추가
    add_current_to_observation: bool = False    # 상태에 모터 전류 추가
    display_cameras: bool = False    # 실행 중 카메라 화면 표시

class ImagePreprocessingConfig:
    crop_params_dict: dict[str, tuple[int, int, int, int]] | None = None    # 이미지 크롭 파라미터
    resize_size: tuple[int, int] | None = None    # 목표 이미지 크기

class GripperConfig:
    use_gripper: bool = True    # 그리퍼 제어 사용
    gripper_penalty: float = 0.0    # 부적절한 그리퍼 사용에 대한 패널티

class ResetConfig:
    fixed_reset_joint_positions: Any | None = None    # 리셋 시 사용할 관절 위치
    reset_time_s: float = 5.0    # 리셋 중 대기 시간
    control_time_s: float = 20.0    # 최대 에피소드 길이
    terminate_on_success: bool = True    # 성공 감지 시 에피소드 종료 여부

class InverseKinematicsConfig:
    urdf_path: str | None = None    # 로봇 URDF 파일 경로
    target_frame_name: str | None = None    # 엔드이펙터 프레임 이름
    end_effector_bounds: dict[str, list[float]] | None = None    # 엔드이펙터 작업 공간 경계
    end_effector_step_sizes: dict[str, float] | None = None    # 축별 엔드이펙터 step 크기

class RewardClassifierConfig:
    pretrained_path: str | None = None    # 사전학습된 보상 분류기 경로
    success_threshold: float = 0.5    # 성공 감지 임곗값
    success_reward: float = 1.0    # 성공한 에피소드의 보상 값

# 데이터셋 설정
class DatasetConfig:
    repo_id: str    # LeRobot 데이터셋 저장소 ID
    task: str    # 작업 식별자
    root: str | None = None    # 로컬 데이터셋 루트 디렉터리
    num_episodes_to_record: int = 5    # 기록할 에피소드 수
    replay_episode: int | None = None    # 재생할 에피소드 인덱스
    push_to_hub: bool = False    # 데이터셋을 Hub에 업로드할지 여부
```

### Processor 파이프라인 아키텍처

HIL-SERL은 모듈형 processor 파이프라인 구조를 사용합니다. 로봇 관측과 액션을 여러 단계로 나누어 처리하며, 각 단계는 조합 가능하도록 설계되어 있습니다.

#### Environment Processor Pipeline

env_processor는 들어오는 관측값과 환경 상태를 처리합니다.

1. **VanillaObservationProcessorStep**: 원시 로봇 관측을 표준 형식으로 변환
2. **JointVelocityProcessorStep** (선택): 관측에 관절 속도 정보 추가
3. **MotorCurrentProcessorStep** (선택): 관측에 모터 전류 정보 추가
4. **ForwardKinematicsJointsToEE** (선택): 관절 위치에서 엔드 이펙터 포즈 계산
5. **ImageCropResizeProcessorStep** (선택): 카메라 이미지 크롭/리사이즈
6. **TimeLimitProcessorStep** (선택): 에피소드 시간 제한 적용
7. **GripperPenaltyProcessorStep** (선택): 부적절한 그리퍼 사용 패널티 적용
8. **RewardClassifierProcessorStep** (선택): 비전 모델 기반 자동 보상 판정
9. **AddBatchDimensionProcessorStep**: 신경망 입력용 배치 차원 추가
10. **DeviceProcessorStep**: 데이터를 지정된 디바이스(CPU/GPU)로 이동

#### Action Processor Pipeline

action_processor는 출력 액션과 사람 개입을 처리합니다.

1. **AddTeleopActionAsComplimentaryDataStep**: 텔레오퍼레이션 액션을 로깅용 보조 데이터로 기록
2. **AddTeleopEventsAsInfoStep**: 개입 이벤트와 에피소드 제어 신호 기록
3. **InterventionActionProcessorStep**: 사람 개입 및 에피소드 종료 처리
4. **Inverse Kinematics Pipeline** (활성 시):
   - **MapDeltaActionToRobotActionStep**: 델타 액션을 로봇 액션 형식으로 변환
  - **EEReferenceAndDelta**: 엔드 이펙터 기준점과 델타 이동량 계산
   - **EEBoundsAndSafety**: 작업공간 안전 경계 적용
  - **InverseKinematicsEEToJoints**: 엔드 이펙터 액션을 관절 타깃으로 변환
   - **GripperVelocityToJoint**: 그리퍼 제어 명령 처리

#### 설정 예시

**기본 관측 처리**

```json
{
  "env": {
    "processor": {
      "observation": {
        "add_joint_velocity_to_observation": true,
        "add_current_to_observation": false,
        "display_cameras": false
      }
    }
  }
}
```

**이미지 처리**

```json
{
  "env": {
    "processor": {
      "image_preprocessing": {
        "crop_params_dict": {
          "observation.images.front": [180, 250, 120, 150],
          "observation.images.side": [180, 207, 180, 200]
        },
        "resize_size": [128, 128]
      }
    }
  }
}
```

**Inverse Kinematics 설정**

```json
{
  "env": {
    "processor": {
      "inverse_kinematics": {
        "urdf_path": "path/to/robot.urdf",
        "target_frame_name": "end_effector",
        "end_effector_bounds": {
          "min": [0.16, -0.08, 0.03],
          "max": [0.24, 0.2, 0.1]
        },
        "end_effector_step_sizes": {
          "x": 0.02,
          "y": 0.02,
          "z": 0.02
        }
      }
    }
  }
}
```

### 고급 관측 처리

HIL-SERL은 정책 성능 향상에 도움이 되는 추가 관측 처리 기능을 제공합니다.

#### 관절 속도 처리

정책에 움직임 정보를 제공하려면 joint velocity 추정을 활성화해주세요.

```json
{
  "env": {
    "processor": {
      "observation": {
        "add_joint_velocity_to_observation": true
      }
    }
  }
}
```

이 설정을 사용하면:

- 연속 관절 위치의 유한차분으로 관절 속도를 추정하고
- 속도 정보를 관측 상태 벡터에 추가하며
- 동적 작업에서 모션 인지가 필요한 정책에 특히 유용합니다.

#### 모터 전류 처리

접촉력이나 부하 상태를 파악하려면 motor current를 관측에 포함해주세요.

```json
{
  "env": {
    "processor": {
      "observation": {
        "add_current_to_observation": true
      }
    }
  }
}
```

이 설정을 사용하면:

- 로봇 제어 시스템에서 모터 전류 값을 읽어와
- 관측 상태 벡터에 추가하고
- 접촉 이벤트, 물체 무게, 기계적 저항 감지에 도움을 줍니다.

#### 관측 처리 조합

아래처럼 여러 기능을 동시에 켤 수 있습니다.

```json
{
  "env": {
    "processor": {
      "observation": {
        "add_joint_velocity_to_observation": true,
        "add_current_to_observation": true,
        "display_cameras": false
      }
    }
  }
}
```

**참고:** 관측 기능을 추가하면 상태 공간 차원이 커집니다. 정책 네트워크 구조 조정이나 추가 데이터 수집이 필요할 수 있습니다.

### 로봇 작업 공간 경계 찾기

시연을 수집하기 전에 로봇의 적절한 동작 경계를 먼저 정해야 합니다.

이 과정은 실제 로봇 학습 문제를 두 가지 측면에서 단순화합니다.
1) 작업에 필요한 영역으로 작동 공간을 제한해 불필요하거나 위험한 탐색을 줄이고,
2) 관절 공간 대신 엔드이펙터 공간에서 학습할 수 있게 합니다.

경험적으로 조작 RL에서 관절 공간 학습은 더 어려운 경우가 많습니다. 어떤 작업은 관절 공간에서는 거의 학습이 어렵지만, 액션 공간을 엔드 이펙터 좌표로 바꾸면 학습 가능해집니다.

**lerobot-find-joint-limits 사용**

이 스크립트는 엔드이펙터의 안전 작업 경계를 찾는 데 도움을 줍니다. follower와 leader arm이 있으면 follower arm의 경계를 측정해 학습 시 적용할 수 있습니다.
액션 공간에 경계를 두면 에이전트의 불필요한 중복 탐색을 줄일 수 있고, 안전성도 확보할 수 있습니다.

```bash
lerobot-find-joint-limits \
  --robot.type=so100_follower \
  --robot.port=/dev/tty.usbmodem58760431541 \
  --robot.id=black \
  --teleop.type=so100_leader \
  --teleop.port=/dev/tty.usbmodem58760431551 \
  --teleop.id=blue
```

**워크플로우**

1. 스크립트를 실행하고, 작업을 해결하는 범위 내에서 로봇을 움직입니다.
2. 스크립트가 엔드 이펙터의 최소/최대 위치와 관절 각도를 기록해 콘솔에 출력합니다.
   ```
   Max ee position [0.2417 0.2012 0.1027]
   Min ee position [0.1663 -0.0823 0.0336]
   Max joint positions [-20.0, -20.0, -20.0, -20.0, -20.0, -20.0]
   Min joint positions [50.0, 50.0, 50.0, 50.0, 50.0, 50.0]
   ```
3. 이 값을 TeleoperatorConfig의 end_effector_bounds에 반영합니다.

**설정 예시**

```json
"end_effector_bounds": {
    "max": [0.24, 0.20, 0.10],
    "min": [0.16, -0.08, 0.03]
}
```

### 시연 데이터 수집

경계를 정했다면 학습용 시연을 안전하게 수집할 수 있습니다. 오프폴리시 RL은 오프라인 데이터셋을 함께 활용할 수 있어 학습 효율을 높이는 데 유리합니다.

**Record 모드 설정**

시연 수집용 설정 파일을 만드세요(또는 기존 env_config.json을 수정).

1. 루트 레벨 mode를 record로 설정
2. dataset 섹션에 고유 repo_id 지정(예: username/task_name)
3. dataset.num_episodes_to_record에 수집 에피소드 수 지정
4. env.processor.image_preprocessing.crop_params_dict를 우선 빈 객체로 설정
5. env 섹션에 env.robot, env.teleop 등 하드웨어 설정 입력

예시:

```json
{
  "env": {
    "type": "gym_manipulator",
    "name": "real_robot",
    "fps": 10,
    "processor": {
      "control_mode": "gamepad",
      "observation": {
        "display_cameras": false
      },
      "image_preprocessing": {
        "crop_params_dict": {},
        "resize_size": [128, 128]
      },
      "gripper": {
        "use_gripper": true,
        "gripper_penalty": 0.0
      },
      "reset": {
        "reset_time_s": 5.0,
        "control_time_s": 20.0
      }
    },
    "robot": {
      // ... robot configuration ...
    },
    "teleop": {
      // ... teleoperator configuration ...
    }
  },
  "dataset": {
    "repo_id": "username/pick_lift_cube",
    "root": null,
    "task": "pick_and_lift",
```

### 텔레오퍼레이션 장치 사용

로봇과 함께 텔레오퍼레이션 장치가 필요합니다. 데이터셋 수집과 온라인 학습 중 개입을 위해 사용합니다.
게임패드, 키보드, 로봇 리더 암을 지원합니다.

HIL-SERL은 로봇 엔드 이펙터 공간에서 액션을 학습하므로, 텔레오퍼레이션 장치는 엔드 이펙터의 x, y, z 변위를 제어합니다.

이를 위해 엔드 이펙터 공간 액션을 받는 로봇 버전을 정의해야 합니다. 기본 파라미터는 SO100FollowerEndEffector와 SO100FollowerEndEffectorConfig를 참고해주세요.

<!-- prettier-ignore-start -->
```python
class SO100FollowerEndEffectorConfig(SO100FollowerConfig):
    """Configuration for the SO100FollowerEndEffector robot."""

    # 엔드이펙터 위치의 기본 경계(미터 단위)
    end_effector_bounds: dict[str, list[float]] = field( # x, y, z 방향 엔드 이펙터 경계
        default_factory=lambda: {
            "min": [-1.0, -1.0, -1.0],  # x, y, z 최소값
            "max": [1.0, 1.0, 1.0],  # x, y, z 최대값
        }
    )

    max_gripper_pos: float = 50 # 그리퍼가 열린 상태로 간주되는 최대 그리퍼 위치

    end_effector_step_sizes: dict[str, float] = field( # x, y, z 방향 엔드 이펙터 최대 step 크기
        default_factory=lambda: {
            "x": 0.02,
            "y": 0.02,
            "z": 0.02,
        }
    )
```
<!-- prettier-ignore-end -->

Teleoperator는 텔레오퍼레이션 장치를 정의합니다. 사용 가능한 장치 목록은 lerobot/teleoperators에서 확인할 수 있습니다.

**시연 녹화 시작**

예시 설정 파일로 녹화를 시작합니다.

```bash

```

녹화 중 동작:

1. 로봇은 env.processor.reset.fixed_reset_joint_positions에 정의된 초기 자세로 리셋됩니다.
2. 작업을 성공적으로 수행합니다.
3. 성공 버튼을 누르면 보상 1로 에피소드가 종료됩니다.
4. 시간 제한 도달 또는 실패 버튼 입력 시 보상 0으로 종료됩니다.
5. rerecord 버튼으로 현재 에피소드를 다시 기록할 수 있습니다.
6. 완료되면 다음 에피소드로 자동 진행됩니다.
7. 모든 에피소드 종료 후 데이터셋은 로컬 저장되고(선택 시) Hub로 업로드됩니다.

### 데이터셋 전처리

시연을 수집한 뒤에는 최적의 카메라 크롭 영역을 찾기 위해 데이터셋을 처리합니다.
RL은 배경 잡음에 민감하므로 작업 영역 위주로 이미지를 자르는 것이 중요합니다.

비전 기반 RL은 픽셀 입력을 직접 학습하기 때문에 조명 변화, 그림자, 작업 외부 움직임 같은 불필요한 시각 정보에 쉽게 영향을 받습니다. 좋은 ROI는 다음을 만족해야 합니다.

- 작업이 실제로 일어나는 핵심 영역만 포함
- 로봇 엔드 이펙터와 작업 물체를 모두 포함
- 불필요한 배경 요소와 시각적 잡음은 제외

참고: 이미 crop 파라미터를 알고 있다면 이 단계는 생략하고 설정 파일의 crop_params_dict에 바로 입력해도 됩니다.

**크롭 파라미터 추출**

3. c 키로 선택을 확정합니다.
4. 모든 카메라에 대해 반복합니다.
5. 스크립트가 크롭 파라미터를 출력하고, 크롭된 새 데이터셋을 만듭니다.

```
observation.images.front: [180, 250, 120, 150]
```

### 보상 분류기 학습

보상 분류기는 HIL-SERL에서 중요한 역할을 합니다. 사람이 매 타임스텝 보상을 지정하거나 수작업 reward function을 설계하지 않아도, 시각 관측으로 성공/실패를 예측해 자동으로 보상을 할당합니다. 이로 인해 RL이 더 안정적이고 효율적으로 학습됩니다.

이 섹션은 LeRobot human-in-the-loop RL에서 보상 분류기를 학습하는 방법을 설명합니다. 보상 분류기는 상태를 입력받아 reward 값을 예측합니다.

**참고:** 보상 분류기 학습은 선택 사항입니다. 초기 실험은 게임패드/키보드로 성공을 수동 라벨링하며 시작해도 됩니다.

modeling_classifier.py 구현은 사전학습 비전 모델로 이미지를 처리합니다. 성공/실패 이진 분류(단일 값) 또는 멀티클래스 출력 모두 지원합니다.

**보상 분류기용 데이터셋 수집**

학습 전에 라벨된 데이터셋을 먼저 수집해야 합니다. gym_manipulator.py의 record_dataset 함수로 관측, 액션, 보상 데이터를 기록할 수 있습니다.

아래처럼 HILSerlRobotEnvConfig 기반 설정 일부를 수정한 뒤 실행합니다.

```bash
python -m lerobot.rl.gym_manipulator --config_path src/lerobot/configs/reward_classifier_train_config.json
```

**데이터 수집 핵심 파라미터**

- mode: 루트 레벨에서 record로 설정
- dataset.repo_id: hf_username/dataset_name 형태의 Hub 저장소 ID
- dataset.num_episodes_to_record: 수집할 에피소드 수
- env.processor.reset.terminate_on_success: 성공 감지 시 자동 종료 여부(기본 true)
- env.fps: 기록 프레임레이트
- dataset.push_to_hub: Hub 업로드 여부

env.processor.reset.terminate_on_success를 false로 두면 성공 이후에도 에피소드가 계속되어 reward=1인 양성 예시를 더 많이 수집할 수 있습니다. 보상 분류기 학습에는 이 설정이 매우 중요합니다. true(기본)일 때는 성공 즉시 에피소드가 종료됩니다.

**중요:** 보상 분류기 데이터 수집 시 terminate_on_success를 false로 설정해 양성 예시를 충분히 모아주세요. 일반 HIL-SERL 학습에서는 true를 유지하는 편이 일반적입니다.

예시:

```json
{
  "env": {
    "type": "gym_manipulator",
    "name": "real_robot",
    "fps": 10,
    "processor": {
      "reset": {
        "reset_time_s": 5.0,
        "control_time_s": 20.0,
        "terminate_on_success": false
      },
      "gripper": {
        "use_gripper": true
      }
    },
    "robot": {
      // ... robot configuration ...
    },
    "teleop": {
      // ... teleoperator configuration ...
    }
  },
  "dataset": {
    "repo_id": "hf_username/dataset_name",
    "dataset_root": "data/your_dataset",
    "task": "reward_classifier_task",
    "num_episodes_to_record": 20,
    "replay_episode": null,
    "push_to_hub": true
  },
  "mode": "record",
  "device": "cpu"
}
```

**보상 분류기 설정**

보상 분류기 설정은 configuration_classifier.py에서 관리합니다. 주요 파라미터는 다음과 같습니다.

- model_name: 기본 모델 아키텍처(주로 helper2424/resnet10 사용)
- model_type: cnn 또는 transformer
- num_cameras: 카메라 입력 개수

- hidden_dim: 은닉 표현 차원
- dropout_rate: 정규화 비율
- learning_rate: 옵티마이저 학습률

### 액터-러너(Actor-Learner)로 학습하기

LeRobot은 분산 액터-러너(actor-learner) 아키텍처를 사용합니다. 로봇 상호작용과 학습 단계를 분리해 서로 블로킹 없이 동시에 실행할 수 있습니다. 액터 서버는 로봇 관측/액션을 처리해 러너로 전송하고, 러너는 gradient descent로 정책을 업데이트한 뒤 액터로 최신 가중치를 주기적으로 전달합니다.

즉, 러너 프로세스와 액터 프로세스 두 개를 실행해야 합니다.

**설정 준비**

학습 설정 파일을 준비해주세요. train config는 lerobot/configs/train.py의 TrainRLServerPipelineConfig를 기반으로 합니다.

1. policy 설정(type=sac, device 등) 구성
2. dataset을 크롭된 데이터셋으로 지정
3. crop 파라미터를 포함한 환경 설정 확인
4. SAC 관련 파라미터(configuration_sac.py) 점검
5. task에 맞는 input_features/output_features인지 policy 설정 검증

**러너(Learner) 시작**

먼저 러너 서버를 실행합니다.

```bash
python -m lerobot.rl.learner --config_path src/lerobot/configs/train_config_hilserl_so100.json
```

러너 역할:

- 정책 네트워크 초기화
- replay buffer 준비
- 액터 통신용 gRPC 서버 오픈
- 전이 데이터 처리 및 정책 업데이트

**액터(Actor) 시작**

다른 터미널에서 같은 설정으로 액터를 실행합니다.

```bash
python -m lerobot.rl.actor --config_path src/lerobot/configs/train_config_hilserl_so100.json
```

액터 역할:

- gRPC로 러너에 연결
- 환경 초기화
- 정책 롤아웃 실행 및 경험 수집
- 전이 데이터를 러너로 전송
- 업데이트된 정책 파라미터 수신

**학습 진행 흐름**

1. 액터가 환경에서 정책을 실행
2. 전이 데이터가 러너로 전송
3. 러너가 데이터를 바탕으로 정책 업데이트
4. 업데이트된 가중치를 액터로 전달
5. 지정한 step 한도까지 반복

**Human in the Loop**

- 학습 효율의 핵심은 적절한 사람 개입입니다. 필요 시 로봇 동작을 교정하고 과제를 완수해 정책 탐색을 돕습니다.
- 개입하려면 게임패드 우측 상단 트리거(또는 키보드 space)를 눌러주세요. 정책 액션이 멈추고 직접 제어할 수 있습니다.
- 좋은 실험은 초반에는 개입이 많고, 정책이 좋아질수록 개입 비율이 점차 줄어드는 형태입니다. wandb에서 intervention rate를 모니터링해주세요.

### 사람 개입 가이드

학습은 개입 전략에 매우 민감합니다. 효과적인 개입 감각을 익히려면 몇 번의 실험이 필요합니다.

- 학습 초반 몇 에피소드는 정책이 충분히 탐색하도록 둡니다.
- 너무 오래 개입하지 말고, 로봇이 경로를 벗어났을 때 짧고 정확하게 교정합니다.
- 정책이 어느 정도 성공하기 시작하면 완벽하지 않더라도 개입을 최소한의 짧은 조작(예: 빠른 그립 보정)으로 줄입니다.

이상적인 패턴은 아래 그림처럼 학습이 진행될수록 intervention rate가 점진적으로 감소하는 것입니다.

### 튜닝 우선순위가 높은 하이퍼파라미터

아래 값들은 학습 안정성과 속도에 큰 영향을 줍니다.

- **temperature_init** (policy.temperature_init): SAC 초기 entropy temperature. 값이 크면 탐색이 증가하고, 작으면 초반 정책이 더 결정적입니다. 시작점으로 1e-2를 권장합니다. 너무 높으면 사람 개입 효과가 줄고 학습이 느려질 수 있습니다.
- **policy_parameters_push_frequency** (policy.actor_learner_config.policy_parameters_push_frequency): 러너가 액터로 가중치를 푸시하는 간격(초). 기본은 4초입니다. 더 신선한 가중치가 필요하면 1~2초로 줄이고(네트워크 트래픽 증가), 연결이 느릴 때만 값을 키워주세요(샘플 효율 저하 가능).
- **storage_device** (policy.storage_device): 러너가 정책 파라미터를 보관하는 디바이스입니다. GPU 메모리에 여유가 있으면 기본 cpu 대신 cuda를 권장합니다. 가중치를 GPU에 유지하면 CPU→GPU 전송 오버헤드가 줄어 러너 업데이트 속도가 크게 향상될 수 있습니다.
