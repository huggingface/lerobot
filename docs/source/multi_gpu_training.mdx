# Multi-GPU Training

LeRobot trains on multiple GPUs through [Hugging Face Accelerate](https://huggingface.co/docs/accelerate), driven entirely by the training config: every parallelism and runtime knob is a `lerobot-train` CLI flag (`--parallelism.*`, `--accelerator.*`) and is recorded in the checkpoint's `train_config.json`, so any run is reproducible from that file alone. You never write an accelerate yaml.

Three data-parallel layouts are supported, selected purely by the parallelism degrees:

| Layout   | What it does                                                                       | Config                                                  |
| -------- | ---------------------------------------------------------------------------------- | ------------------------------------------------------- |
| **DDP**  | Replicates the full model on every GPU                                             | default on any multi-GPU launch                         |
| **FSDP** | Shards parameters, gradients, and optimizer state across GPUs (FSDP2, ZeRO-3-like) | `--parallelism.dp_shard=N`                              |
| **HSDP** | Shards within groups of GPUs, replicates across groups                             | `--parallelism.dp_replicate=R --parallelism.dp_shard=S` |

## Installation

`accelerate` is included in the `training` extra:

```bash
pip install 'lerobot[training]'
```

## Launching

Accelerate is now used only as a _launcher_: it no longer manages the training configuration — every distributed setting lives in LeRobot's own parameter system (`--parallelism.*`, `--accelerator.*`) and is applied internally. Since a launcher only spawns the processes and sets the rendezvous environment, `torchrun` and `accelerate launch` are interchangeable.

With `torchrun`:

```bash
torchrun --nproc-per-node=2 $(which lerobot-train) \
  --dataset.repo_id=${HF_USER}/my_dataset \
  --policy.type=act \
  --policy.repo_id=${HF_USER}/my_trained_policy \
  --output_dir=outputs/train/act_multi_gpu \
  --job_name=act_multi_gpu \
  --wandb.enable=true
```

With `accelerate launch` (as a plain launcher — no `--config_file`):

```bash
accelerate launch --num_processes=2 $(which lerobot-train) \
  --dataset.repo_id=${HF_USER}/my_dataset \
  --policy.type=act \
  --policy.repo_id=${HF_USER}/my_trained_policy \
  --output_dir=outputs/train/act_multi_gpu \
  --job_name=act_multi_gpu \
  --wandb.enable=true
```

With no `--parallelism.*` flags, a multi-process launch runs plain DDP — the same out-of-the-box behavior as before. Multi-node runs use the standard `torchrun --nnodes/--node-rank/--rdzv-endpoint` flags (or their `accelerate launch --num_machines/--machine_rank/--main_process_ip` equivalents).

> [!WARNING]
> The `accelerate launch --config_file some.yaml` / `accelerate config` flow is superseded. Accelerate yaml files configure the engine through environment variables behind the config system's back, which would make `train_config.json` lie about what actually ran — so `lerobot-train` refuses to start when such variables (`FSDP_*`, `PARALLELISM_CONFIG_*`, ...) are set, and the error names the offending variables. Move the settings to `--parallelism.*` / `--accelerator.*` flags, or set `LEROBOT_ALLOW_ACCELERATE_ENV=1` to acknowledge the override and proceed anyway.

## Batch semantics, learning rate, and steps

Each of the `dp_replicate × dp_shard` data-parallel workers loads its own `--batch_size` micro-batch every step, so one training step consumes `batch_size × dp_world_size` samples — and `× gradient_accumulation_steps` of those go into each optimizer update:

```
effective_batch_size = batch_size × dp_world_size × gradient_accumulation_steps
```

The training banner prints this factorization at startup. `--steps` counts loop steps (micro-batches per worker), not optimizer updates.

Gradient accumulation is a first-class flag:

```bash
torchrun --nproc-per-node=2 $(which lerobot-train) \
  --batch_size=8 --accelerator.gradient_accumulation.steps=4 ...
```

**LeRobot does not auto-scale the learning rate or the number of steps** when the effective batch size grows — you keep full control. If you scale out and want equivalent training, adjust manually, e.g. with 2 GPUs: double `--optimizer.lr` (linear scaling) and/or halve `--steps`.

## Sharded training (FSDP)

DDP replicates the full model, so a model that doesn't fit on one GPU won't fit under DDP either. Shard it instead with FSDP2:

```bash
torchrun --nproc-per-node=4 $(which lerobot-train) \
  --dataset.repo_id=${HF_USER}/my_dataset \
  --policy.type=<your_policy> \
  --parallelism.dp_shard=4 \
  --accelerator.mixed_precision=bf16 \
  --output_dir=outputs/train/my_policy_fsdp
```

`--parallelism.dp_shard=-1` shards over however many processes the launcher started.

### Wrap units

FSDP shards the model in units — typically the repeated transformer block — and gathers one unit at a time during forward/backward. Policies declare their wrap units on the policy class via `_fsdp_wrap_modules` — currently ACT (`["ACTEncoderLayer", "ACTDecoderLayer"]`) and FastWAM (`["MoTLayer"]`) ship declarations, so they shard with no extra flags; for other policies pass one of the flags below. You can override the declaration, or use a size-based policy instead:

```bash
--accelerator.fsdp.wrap_modules='["MyTransformerBlock"]'   # explicit class names
--accelerator.fsdp.min_num_params=1000000                  # or: wrap every submodule above 1M params
```

If a policy declares nothing and you pass neither flag, the run fails at startup rather than silently wrapping only the root module (which would forfeit all sharding memory savings).

Other sharding knobs: `--accelerator.fsdp.reshard_after_forward` (default `true`; set `false` to trade memory for one gather less per step), `--accelerator.fsdp.cpu_offload`, and `--accelerator.fsdp.ignored_modules` (a regex of module paths to keep unsharded).

### Mixed precision

`--accelerator.mixed_precision=bf16` selects torch AMP under DDP and the FSDP2 `MixedPrecisionPolicy` under sharding. Sharded runs accept `no` and `bf16` only (fp16's GradScaler path is not supported over sharded tensors, and the config validation says so up front). Note `--policy.use_amp` only affects standalone inference scripts such as `lerobot-eval`, and `--policy.dtype` only casts the model parameters — neither enables training autocast. Training precision is owned exclusively by `--accelerator.mixed_precision`: set it explicitly (e.g. together with `--policy.dtype=bfloat16`, pass `--accelerator.mixed_precision=bf16`).

### HSDP

At larger scale, sharding across _all_ GPUs makes every gather cross the slow inter-node links. HSDP shards within a replica group (e.g. one node, over NVLink) and replicates across groups (DDP-style gradient sync over the slower links):

```bash
# 16 GPUs = 2 nodes × 8: shard within each node, replicate across nodes
torchrun --nnodes=2 --nproc-per-node=8 ... $(which lerobot-train) \
  --parallelism.dp_replicate=2 --parallelism.dp_shard=8 ...
```

The degrees must multiply to the world size (`dp_replicate × dp_shard = WORLD_SIZE`); the error message echoes every degree when they don't.

## Checkpoints

Every checkpoint contains a `pretrained_model/` directory (the loadable policy: `config.json`, `train_config.json`, processor files, weights) and a `training_state/` directory (step, RNG, optimizer, scheduler). What the _weights_ look like is controlled by `--checkpoint_format`:

| `--checkpoint_format`     | Weights artifact                                 | Use when                                                                   |
| ------------------------- | ------------------------------------------------ | -------------------------------------------------------------------------- |
| `safetensors` _(default)_ | single `model.safetensors` (full gather on save) | you want every checkpoint immediately loadable with `from_pretrained`      |
| `dcp`                     | `pytorch_model_fsdp_0/` shard directory          | full gathers are too slow/large at your scale; convert offline when needed |
| `safetensors_dcp`         | both                                             | you want fast sharded resume _and_ distributable checkpoints               |

The `dcp` and `safetensors_dcp` formats require a sharded run. On sharded runs the optimizer state is always saved as DCP shards (`training_state/optimizer_0/`) — resharding on load is the only correct way to restore it.

Two things to know about gathered (`safetensors`) checkpoints from sharded runs:

- **They store fp32 weights.** Under `bf16` FSDP keeps an fp32 master copy and the checkpoint saves it (~2× the bf16 size) so training resumes consistently; `from_pretrained` casts back to the policy dtype on load. Loading an fp32 checkpoint materializes it in full precision before casting, so on a tight GPU load on CPU first or cast the file offline.
- The gather is collective (all ranks participate) but only the main process writes.

### Converting DCP checkpoints

`lerobot-convert-dcp` merges a DCP shard directory into a regular `model.safetensors`, offline and without GPUs:

```bash
lerobot-convert-dcp --checkpoint_dir=outputs/train/run/checkpoints/005000
lerobot-convert-dcp --checkpoint_dir=... --delete_dcp=true --push_to_hub=${HF_USER}/my_policy
```

`--push_to_hub` publishes the converted directory as a model repo (DCP shards are never uploaded — published repos carry safetensors only).

### Resuming

Resume with `--resume=true --config_path=.../checkpoints/last/pretrained_model/train_config.json`, as on a single GPU. Both model formats resume; DCP resume reshards the model and optimizer state to the _current_ topology, so you can resume on a different number of GPUs or even a different `dp_replicate/dp_shard` split. If a `dcp`-format checkpoint is missing its shard directory (e.g. it was pruned after conversion), the error tells you to run `lerobot-convert-dcp` on what remains or resume from a safetensors checkpoint.

The data sampler resumes at the right epoch and offset in all cases, but is only _sample-exact_ when the world size and batch size match the original run (a warning is logged otherwise); model and optimizer state are unaffected.

> [!NOTE]
> FSDP checkpoints written by lerobot ≤ 0.6.x used a different on-disk layout (a gathered full optimizer state) and are **not resumable** with this version — pin `lerobot==0.6.x` to finish those runs. Their `model.safetensors` remains loadable with `from_pretrained` as usual.

## Notes

- Checkpoint saves and end-of-training publishes are collective: every rank enters them (sharded weights are gathered across ranks), and only the main process writes files, uploads, or logs — so call sites still need no rank branches. wandb and console logging are main-process-only.
- Metrics are reduced across ranks before logging: losses are averaged, timing takes the slowest rank (which is what actually gates the next step), and `samples/s` reports cluster-wide throughput. The gradient norm is not reduced — after gradient sync it is identical on every rank.
- Learning-rate scheduling is stepped once per training step regardless of the number of processes (`step_scheduler_with_optimizer=False` is baked in).
- When saving or publishing, LeRobot unwraps the model from accelerate's wrapper and, if sharded, gathers the full state dict — published models and safetensors-bearing checkpoints load with plain `from_pretrained` (a `dcp`-format checkpoint first needs `lerobot-convert-dcp`, per the table above).

For background on the underlying machinery, see the [Accelerate FSDP guide](https://huggingface.co/docs/accelerate/usage_guides/fsdp). To go deeper on large-scale training, check out the [Ultrascale Playbook](https://huggingface.co/spaces/nanotron/ultrascale-playbook).
