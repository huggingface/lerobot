# Isaac Teleop

Control your robot with NVIDIA [Isaac Teleop](https://github.com/NVIDIA/IsaacTeleop), a
multi-modal teleoperation framework. Isaac Teleop drives a single `TeleopSession` from a range
of input devices — XR (VR) controllers, hand tracking, full-body tracking, Manus gloves, foot
pedals, and more.

In LeRobot, each Isaac Teleop input device is its own `Teleoperator` subclass under
`lerobot.teleoperators.isaac_teleop`, sharing one session lifecycle (see
`IsaacTeleopTeleoperator`). The device available today is the **XR controller**
(`XRController`); Manus gloves and hand/full-body tracking are the natural next devices. This
guide uses the XR controller.

**In this guide you'll learn:**

- How an Isaac Teleop device drives a robot end‑effector (EE) target
- How the _clutch_ (squeeze/grip on the XR controller) engages teleoperation without jerking the arm
- How to run the SO‑101 teleoperation example and tune motion / gripper / IK

## Installation

Isaac Teleop support is the `isaac-teleop` extra. `isaacteleop` lives on the NVIDIA package index
(not public PyPI) and 1.3.x ships as a pre-release, so point your installer at the NVIDIA index
and allow pre-releases. Both forms below pull `isaacteleop[cloudxr,retargeters]~=1.3.0` — the
`cloudxr` extra brings the CloudXR runtime bindings and `retargeters` the retargeter library the
controller pipeline uses.

### Into an existing LeRobot install

```bash
uv pip install 'lerobot[isaac-teleop]' \
  --extra-index-url https://pypi.nvidia.com --prerelease=allow
```

### From LeRobot source

If you build LeRobot from source (e.g. to work on the teleoperator itself), clone the repo and
install the extra in editable mode. The NVIDIA index is already configured in `pyproject.toml`
(`[[tool.uv.index]]`), so the explicit `--extra-index-url` is optional — but `--prerelease=allow`
is still required because 1.3.x is a pre-release:

```bash
git clone https://github.com/huggingface/lerobot.git
cd lerobot
uv pip install -e '.[isaac-teleop]' --prerelease=allow
```

`uv sync --locked` does **not** install this extra — the private pre-release is intentionally
left out of the lockfile — so use the explicit `uv pip install -e` above. You can also install
the underlying package on its own, which is handy when developing against a local Isaac Teleop
checkout (see [Build Isaac Teleop from source](https://nvidia.github.io/IsaacTeleop/main/getting_started/build_from_source/index.html)):

```bash
uv pip install 'isaacteleop[cloudxr,retargeters]~=1.3.0' \
  --extra-index-url https://pypi.nvidia.com --prerelease=allow
```

<Tip warning={true}>

`isaacteleop` is distributed by NVIDIA and is **not** on public PyPI — `pip install
lerobot[isaac-teleop]` without the NVIDIA index will fail. Because the package is private and
pre-release, the `isaac-teleop` extra is intentionally not pinned in the public lockfile.

</Tip>

### Set up CloudXR and connect a headset

Isaac Teleop streams the headset to your machine over **NVIDIA CloudXR**, which provides the
OpenXR runtime the session connects to. By default LeTeleop **auto-launches the CloudXR runtime
for you** when you call `teleop_device.connect()` — you no longer have to run `python -m
isaacteleop.cloudxr` and `source cloudxr.env` in a separate shell. All you need is a supported
headset connected and the CloudXR firewall ports open. Follow the Isaac Teleop
[Quick Start](https://nvidia.github.io/IsaacTeleop/main/getting_started/quick_start.html) for the
headset-pairing and firewall details.

**First run (EULA).** The very first launch must accept the NVIDIA CloudXR EULA. The auto-launch
prompts for it **on stdin**, so on a headless machine it will hang waiting for input. Bootstrap
the EULA once, interactively, with:

```bash
python -m isaacteleop.cloudxr --accept-eula   # one-time: accept the CloudXR EULA
```

After that, `connect()` launches the runtime non-interactively. The launch **blocks for ~30s**
while the runtime comes up.

**Configuration.** Two fields on `IsaacTeleopConfig` (shared by every device) control this:

- `auto_launch_cloudxr` (default `True`) — whether `connect()` starts the runtime. Set `False`
  when CloudXR is already running externally.
- `cloudxr_env_file` (default `None`) — an optional CloudXR device-profile `.env` selecting the
  headset transport (e.g. an Apple Vision Pro profile). This is launcher **input**; it is not the
  `~/.cloudxr/run/cloudxr.env` **output** file the old manual flow told you to `source`. `None`
  keeps the default auto-WebRTC profile.

**Opting out.** To skip the auto-launch (CloudXR already running), either set
`auto_launch_cloudxr=False` or export:

```bash
export LEROBOT_CLOUDXR_SKIP_AUTOLAUNCH=1
```

The **env var takes precedence over the config field**: if `LEROBOT_CLOUDXR_SKIP_AUTOLAUNCH=1` is
set, the auto-launch is skipped even when `auto_launch_cloudxr=True`. This variable is
**independent** of Isaac Lab's `ISAACLAB_CXR_SKIP_AUTOLAUNCH` — setting one does not affect the
other.

**One teleoperator per process.** The CloudXR runtime configures the environment process-wide (a
singleton), so run a single Isaac Teleop teleoperator per process.

**Shutting down.** Always call `teleop_device.disconnect()` on exit — including on Ctrl-C. Wrap
your teleoperation loop in `try/finally` and call `disconnect()` in the `finally`. This tears down
the OpenXR session **before** the CloudXR runtime, which is the required order; the launcher's
`atexit` hook only reaps the runtime and does not run the session's `__exit__`, so without an
explicit `disconnect()` an interrupted run shuts down in the wrong order.

```python
teleop_device.connect()
try:
    while True:
        action = teleop_device.get_action()
        # ... drive the robot ...
finally:
    teleop_device.disconnect()
```

See [System Requirements](https://nvidia.github.io/IsaacTeleop/main/references/requirements.html)
for supported OS / GPU / CloudXR versions and headsets.

## How it works

The XR controller is one Isaac Teleop **input** device. Inside `XRController` it drives three
Isaac Teleop SO‑101 retargeters that turn a single controller into an already clutch‑rebased
**absolute** EE pose, a wrist‑roll angle, an absolute wrist‑pitch angle, and an analog gripper
closedness. That output then flows through LeRobot's existing closed‑loop Cartesian IK pipeline —
the same one the phone teleoperator uses. The only device‑specific piece is the first two stages
(`XRController` + `MapXRControllerActionToRobotAction`); everything downstream is shared, and a
future device (e.g. Manus gloves) would swap in its own `teleop_<device>.py` + processor while
reusing the rest.

`XRController._build_pipeline` wires Isaac Teleop's `ControllersSource` — statically rebased into
the robot base frame by the native `ControllerTransform` (`base_T_anchor`) — into the three SO‑101
retargeters:

- `SO101ClutchRetargeter` → a 7D absolute `ee_pose` (position + grip quaternion) in the base frame.
  It owns the clutch: it latches its engage origin on the session RUNNING edge and rebases motion
  around it, so engaging does not teleport the arm.
- `SO101WristRetargeter` → a `roll_command` (engage‑relative swing‑twist about the controller's
  local Z) and a `pitch_command` (absolute world‑elevation of the controller's AIM/pointer ray),
  both in radians.
- `SO101GripperRetargeter` → a `gripper_command` closedness in `[0, 1]` (analog, proportional to
  the trigger).

See the Isaac Teleop
[Retargeting interface](https://nvidia.github.io/IsaacTeleop/main/references/retargeting/index.html)
and [architecture overview](https://nvidia.github.io/IsaacTeleop/main/overview/architecture.html)
for how those source nodes and retargeters compose.

```text
  VR controller (OpenXR)
        │
        ▼
  XRController.get_action()                  ─┐  device-specific (Isaac Teleop)
        │  ee_pose / wrist_roll / wrist_pitch  │  • TeleopSession; squeeze → enabled (RUNNING)
        │  closedness / enabled                │  • 3 SO-101 retargeters; clutch owns the
        ▼                                       │    engage latch + no-teleport rebasing
  MapXRControllerActionToRobotAction         ─┘  • absolute ee.x/y/z; ee.w* = pitch target;
        │  ee.x/y/z / ee.w* / ee.gripper_pos        ee.gripper_pos = (1 - closedness) * 100;
        │  wrist_roll                               wrist_roll carried through
        ▼
  EEBoundsAndSafety                ── workspace clip + per-frame step clamp (clamp+warn)
        │
        ▼
  InverseKinematicsEEToJoints      ── closed-loop Placo IK; position + soft-pitch
        │  (orientation_weight=0.1)    (passes ee.gripper_pos → gripper.pos)
        ▼
  OverwriteWristRollFromAngle      ── wrist_roll [rad] → wrist_roll.pos [deg] (post-IK)
        │
        ▼
  SO-101 follower joint targets
```

### The clutch: owned by the retargeter

Unlike the phone pipeline (which splits the clutch across `MapPhoneActionToRobotAction` and
`EEReferenceAndDelta`), the XR clutch lives entirely inside `SO101ClutchRetargeter`. It emits an
**absolute** EE pose, so there is no `EEReferenceAndDelta` stage and no delta accumulation on the
LeRobot side — `MapXRControllerActionToRobotAction` is a pure, stateless per‑frame mapping.

The retargeter latches its engage origin on the session **RUNNING edge** (the moment the squeeze
crosses `clutch_threshold`) and drives the EE from the motion *relative* to that origin, so the
arm does not teleport on engage. It seeds its reset‑origin home from the `home_base_T_ee`
transform the example computes by forward kinematics on the arm's **measured joints at connect** —
so the seeded home equals where the arm already is, the loop never commands the arm at startup, and
the first engage is jump‑free. On a mid‑task re‑clutch it resumes from the last commanded pose.

## Controls

- **Squeeze / grip** — the **clutch** (deadman). Hold it past `clutch_threshold` to engage
  teleoperation; release to pause. Each engage re‑captures the origin, so you can reposition
  your hand while paused and re‑engage without the arm jumping (index/clutch style).
- **Trigger** — the **gripper**, controlled **analog**. The jaw tracks the trigger
  proportionally — a half‑pressed trigger leaves the jaw half‑closed — via a closedness in
  `[0, 1]` (0 = open, 1 = closed) that maps to an absolute gripper joint target.
- **Controller orientation** — **wrist roll and pitch**. Roll is the engage‑relative twist about
  the controller's own axis (driven onto `wrist_roll` post‑IK); pitch is the absolute elevation
  of the controller's aim ray (a soft orientation target the IK tracks alongside position).

## Get started

### Step 1: Create the teleoperator

```python
from lerobot.teleoperators.isaac_teleop import XRController, XRControllerConfig

teleop_config = XRControllerConfig(
    hand_side="right",      # "left" or "right" controller
    clutch_threshold=0.5,   # squeeze value above which the clutch engages
)
teleop_device = XRController(teleop_config)
```

`XRController.get_action()` returns the clutch‑rebased absolute `ee_pose` (7,) —
`[x, y, z, qx, qy, qz, qw]` in the robot base frame — plus scalar `wrist_roll` [rad],
`wrist_pitch` [rad], `closedness` (`[0, 1]`, 0 = open), and the boolean `enabled` clutch.

### Step 2: Connect

Calling `teleop_device.connect()` first auto-launches the CloudXR runtime (unless you opted out —
see [Set up CloudXR and connect a headset](#set-up-cloudxr-and-connect-a-headset); this blocks for
~30s and on the first run prompts for the EULA on stdin), then starts the Isaac Teleop
[`TeleopSession`](https://nvidia.github.io/IsaacTeleop/main/getting_started/teleop_session.html)
(opens the OpenXR session and discovers the controllers). XR controllers are self‑calibrating, so
there is no manual calibration step — the clutch handles re‑centering each time you engage. Pair
`connect()` with a `try/finally` that calls `disconnect()` so the session tears down before the
runtime on exit/Ctrl-C.

### Step 3: Run the example

The example assumes you configured your robot (SO‑101 follower) and set the correct serial port.

You also need the **robot URDF (and its meshes) in the example folder**. The example loads
`./SO101/so101_new_calib.urdf` relative to its own directory, so run the example from there. A
helper script downloads the URDF and the meshes it references from the
[SO‑ARM100 repo](https://github.com/TheRobotStudio/SO-ARM100/tree/main/Simulation/SO101) into
`examples/isaac_teleop_to_so101/SO101/`:

```bash
cd examples/isaac_teleop_to_so101
python download_assets.py    # fetches SO101/so101_new_calib.urdf + assets/*.stl
python teleoperate.py
```

`download_assets.py` uses only the standard library (no extra dependencies) and skips files
already present; pass `--force` to re‑download. If you prefer, you can instead copy the `SO101`
folder from the SO‑ARM100 repo into `examples/isaac_teleop_to_so101/` by hand — the script just
automates that, fetching only the meshes the URDF actually references.

Then, in your headset: squeeze and hold the grip to engage, move the controller to drive the
arm, twist/tilt it to roll and pitch the wrist, and press the trigger to close the gripper
(proportionally — release to open).

## Important pipeline steps and options

The clutch retargeter already rebases the pose into the robot base frame, so the LeRobot side is a
thin **absolute‑pose** path — there is no frame remap, no delta accumulation, and no
`EEReferenceAndDelta` stage.

- `MapXRControllerActionToRobotAction` is a stateless per‑frame mapping from the device output to
  the IK input contract. It writes the absolute base‑frame position, encodes the absolute
  wrist pitch as an orientation target, inverts the closedness into a motor gripper target, and
  carries the roll through for the post‑IK overwrite:

  ```python
  action["ee.x"], action["ee.y"], action["ee.z"] = ee_pose[:3]   # absolute, base frame [m]
  action["ee.wx"], action["ee.wy"], action["ee.wz"] = pitch_rotvec  # pitch target (about base Y)
  action["ee.gripper_pos"] = (1 - closedness) * 100   # motor units; SO-101 calibrates 100 = open
  action["wrist_roll"] = wrist_roll                   # [rad], for OverwriteWristRollFromAngle
  ```

  The gripper polarity (`100 = open, 0 = closed`) and the pitch axis/sign are hardware‑calibration
  conventions in the source — flip them there if the jaw opens
  when it should close or the wrist tilts the wrong way.

- `EEBoundsAndSafety` clamps the EE to a workspace and rate‑limits per‑frame jumps. The clutch's
  no‑teleport keeps frames small, so `max_ee_step_m` mostly catches transient controller tracking
  glitches. Set `raise_on_jump=False` so an over‑limit frame is **clamped and warned** instead of
  raising — a crash mid‑loop would leave the arm uncontrolled:

  ```python
  EEBoundsAndSafety(
      end_effector_bounds={"min": [-1.0, -1.0, -1.0], "max": [1.0, 1.0, 1.0]},
      max_ee_step_m=0.10,
      raise_on_jump=False,
  )
  ```

- `InverseKinematicsEEToJoints(initial_guess_current_joints=True, orientation_weight=0.1)` solves
  closed‑loop Placo IK seeded from the measured joints each frame. SO‑101 is a 5‑DOF arm, so the
  IK is position‑dominant; the small `orientation_weight` lets it softly track the wrist‑pitch
  target carried in `ee.w*` (the LeRobot analogue of Isaac Lab's reduced 3‑position + 1‑pitch IK),
  while the terminal roll stays under‑determined and is recovered separately below. There is **no**
  `GripperVelocityToJoint`: the absolute `ee.gripper_pos` is passed straight to `gripper.pos`.
  `initial_guess_current_joints=True` is required so the post‑IK roll overwrite does not drift the
  IK seed. Tune `orientation_weight` on hardware — too high fights position tracking, too low
  ignores the pitch command.

- `OverwriteWristRollFromAngle` runs **after** IK and writes the operator's wrist roll
  (`wrist_roll` [rad] → `wrist_roll.pos` [deg]), overriding the under‑determined roll the
  position IK produces on the 5‑DOF arm.

The example also gates safety at the loop level: it commands the robot **only while the clutch is
engaged**, and re‑sends the measured joints while disengaged, so launching the script never moves
the arm and releasing the clutch freezes it in place.

See the [Processors for Robots and Teleoperators](./processors_robots_teleop) guide for more on
adapting the pipeline to other robots.

## Troubleshooting

- **`ModuleNotFoundError: isaacteleop`** — the `isaac-teleop` extra was installed without the
  NVIDIA index. Re-run the install command at the top of this guide, with `--extra-index-url
https://pypi.nvidia.com --prerelease=allow`.
- **No controllers found** — make sure the CloudXR runtime is running, the firewall ports are
  whitelisted, and the headset is connected (see
  [Set up CloudXR and connect a headset](#set-up-cloudxr-and-connect-a-headset) and the Isaac
  Teleop [Quick Start](https://nvidia.github.io/IsaacTeleop/main/getting_started/quick_start.html)).
- **CloudXR auto-launch failed** — `connect()` raises a `RuntimeError` if the runtime does not
  come up within its startup timeout. Check the launcher logs under `~/.cloudxr/logs`. Common
  causes: the EULA was never accepted (run `python -m isaacteleop.cloudxr --accept-eula` once,
  interactively — the auto-launch prompts on stdin and hangs headless), or the runtime is already
  running externally (set `LEROBOT_CLOUDXR_SKIP_AUTOLAUNCH=1` or `auto_launch_cloudxr=False` to
  skip the auto-launch).
- **Arm does not move** — the clutch is a deadman: you must hold the squeeze/grip past
  `clutch_threshold`. Lower the threshold if your controller's squeeze is reported softly.
- **Motion feels misaligned** — confirm the headset/play space orientation. The controller stream
  is rebased into the robot base frame by the `base_T_anchor` transform on `XRControllerConfig`
  (default: standard OpenXR → robot axis convention); adjust it if your anchor frame differs.
