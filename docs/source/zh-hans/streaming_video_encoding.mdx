# 流式视频编码指南

## 1. 概述

流式视频编码消除了视频数据集录制过程中传统的 PNG 中转步骤。原有流程为：

1. 采集帧 -> 将 PNG 写入磁盘 -> （片段结束时）读取 PNG -> 编码为 MP4 -> 删除 PNG

现在可以在采集过程中实时编码：

1. 采集帧 -> 送入编码器线程队列 -> 直接编码为 MP4

这使得 `save_episode()` 几乎瞬时完成（视频在片段结束时已编码完毕），并消除了片段之间原本存在的阻塞等待，尤其是在多摄像头、长片段场景下。

## 2. 调参参数

| 参数                    | CLI 标志                           | 类型          | 默认值        | 描述                                                        |
| ----------------------- | ---------------------------------- | ------------- | ------------- | ----------------------------------------------------------- |
| `streaming_encoding`    | `--dataset.streaming_encoding`     | `bool`        | `True`        | 在采集过程中启用实时编码                                    |
| `vcodec`                | `--dataset.vcodec`                 | `str`         | `"libsvtav1"` | 视频编解码器。`"auto"` 自动检测最佳硬件编码器               |
| `encoder_threads`       | `--dataset.encoder_threads`        | `int \| None` | `None`（自动）| 每个编码器实例的线程数。`None` 由编解码器自行决定           |
| `encoder_queue_maxsize` | `--dataset.encoder_queue_maxsize`  | `int`         | `60`          | 每个摄像头的最大缓冲帧数（30fps 下约 2 秒），消耗 RAM       |

## 3. 性能注意事项

流式编码意味着 CPU 在采集循环**过程中**编码视频，而非事后处理。这产生了一个 CPU 预算，需要在以下任务之间共享：

- **控制循环**（读取摄像头、控制机器人、写入非视频数据）
- **编码器线程**（每个摄像头一个线程池）
- **Rerun 可视化**（如启用）
- **操作系统及其他进程**

### 分辨率与摄像头数量的影响

| 配置                         | 吞吐量（像素/秒） | CPU 编码负载 | 备注                         |
| ---------------------------- | ----------------- | ------------ | ---------------------------- |
| 2摄像头×640x480x3 @30fps     | 55M               | 低           | 大多数系统可正常运行         |
| 2摄像头×1280x720x3 @30fps    | 165M              | 中等         | 现代系统可轻松应对           |
| 2摄像头×1920x1080x3 @30fps   | 373M              | 高           | 需要高性能 CPU               |

### `encoder_threads` 调优

此参数控制每个编码器实例内部使用的线程数：

- **较高值**（如 4-5）：编码更快，但每个摄像头占用更多 CPU 核心。适合核心数多的高端系统。
- **较低值**（如 1-2）：每个摄像头占用 CPU 更少，为采集和可视化释放更多核心。适合低分辨率图像和性能较强的 CPU。
- **`None`（默认）**：由编解码器自行决定，相关信息可在编解码器日志中查看。

### 背压与丢帧

每个摄像头有一个有界队列（`encoder_queue_maxsize`，默认 60 帧）。当编码器跟不上时：

1. 队列填满（消耗 RAM）
2. 新帧被**丢弃**（不阻塞）——采集循环继续不中断
3. 记录警告日志：`"Encoder queue full for {camera}, dropped N frame(s)"`
4. 片段结束时，报告每个摄像头的总丢帧数

### 编码器跟不上的症状

- **系统感觉卡顿或卡死**：所有 CPU 占用率达到 100%
- **日志中出现丢帧警告**，或录制数据集中的帧数/帧率低于预期
- **机器人运动不流畅**：若 CPU 严重过载，甚至采集循环也可能受影响
- **Rerun 可视化滞后累积**：可视化落后于实时状态

## 4. 硬件加速编码

### 何时使用

在以下情况下使用硬件编码：

- CPU 成为瓶颈（丢帧、机器人运动不流畅、Rerun 滞后）
- 拥有兼容硬件（GPU 或专用编码器）
- 以高吞吐量录制（高分辨率或多摄像头）

### 选择编解码器

| 编解码器              | CPU 占用 | 文件大小        | 质量 | 备注                                               |
| --------------------- | -------- | --------------- | ---- | -------------------------------------------------- |
| `libsvtav1`（默认）   | 高       | 最小            | 最佳 | 默认值。压缩率最高但 CPU 占用最高                  |
| `h264`                | 中等     | 大约大 30-50%   | 良好 | 软件 H.264，CPU 占用较低                           |
| 硬件编码器            | 极低     | 最大            | 良好 | 卸载至专用硬件，适合 CPU 受限系统                  |

### 可用硬件编码器

| 编码器              | 平台          | 硬件                                                                               | CLI 值                                 |
| ------------------- | ------------- | ---------------------------------------------------------------------------------- | -------------------------------------- |
| `h264_videotoolbox` | macOS         | Apple Silicon / Intel                                                              | `--dataset.vcodec=h264_videotoolbox`   |
| `hevc_videotoolbox` | macOS         | Apple Silicon / Intel                                                              | `--dataset.vcodec=hevc_videotoolbox`   |
| `h264_nvenc`        | Linux/Windows | NVIDIA GPU                                                                         | `--dataset.vcodec=h264_nvenc`          |
| `hevc_nvenc`        | Linux/Windows | NVIDIA GPU                                                                         | `--dataset.vcodec=hevc_nvenc`          |
| `h264_vaapi`        | Linux         | Intel/AMD GPU                                                                      | `--dataset.vcodec=h264_vaapi`          |
| `h264_qsv`          | Linux/Windows | Intel Quick Sync                                                                   | `--dataset.vcodec=h264_qsv`            |
| `auto`              | 任意          | 自动探测系统可用的硬件编码器，若未找到则回退到 `libsvtav1`                         | `--dataset.vcodec=auto`                |

> [!NOTE]
> 使用硬件加速编码器可能需要升级 GPU 驱动程序。

> [!NOTE]
> `libsvtav1` 为默认值，因为它提供最佳的训练性能；其他编解码器可降低 CPU 占用并提升速度，但通常会产生更大的文件，且可能影响训练时间。

## 5. 故障排除

| 症状                                                 | 可能原因                         | 解决方案                                                                                                                                                                                               |
| ---------------------------------------------------- | -------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 系统卡死、机器人运动不流畅或 Rerun 可视化滞后        | CPU 不足（占用率 100%）          | 关闭其他应用，降低编码吞吐量，减小 `encoder_threads`，使用 `h264`，设置 `display_data=False`。若 CPU 持续 100%，说明配置超出系统能力，考虑使用 `--dataset.streaming_encoding=false` 或硬件编码（`--dataset.vcodec=auto`） |
| "Encoder queue full"警告或数据集中存在丢帧            | 编码器跟不上（队列溢出）         | 若 CPU 未达 100%：增大 `encoder_threads`，增大 `encoder_queue_maxsize`，或使用硬件编码（`--dataset.vcodec=auto`）                                                                                       |
| RAM 占用过高                                          | 队列填充速度快于编码速度         | `encoder_threads` 过低或 CPU 不足，减小 `encoder_queue_maxsize` 或使用硬件编码                                                                                                                         |
| 视频文件过大                                          | 使用了硬件编码器或 H.264         | 预期的权衡，若 CPU 允许可切换回 `libsvtav1`                                                                                                                                                            |
| `save_episode()` 仍然很慢                             | `streaming_encoding` 为 `False`  | 设置 `--dataset.streaming_encoding=true`                                                                                                                                                               |
| 编码器线程崩溃                                        | 编解码器不可用或设置无效         | 检查 `vcodec` 是否已安装，尝试 `--dataset.vcodec=auto`                                                                                                                                                 |
| 录制的数据集缺少帧                                    | CPU/GPU 不足或偶发性负载峰值     | 若约 5% 的帧缺失，说明系统过载，请按上述建议操作；若缺失帧较少（约 2%），可能是启动时的偶发性瞬时负载峰值所致，属正常现象                                                                              |

## 6. 推荐配置

以下估算较为保守，建议在实际配置上进行测试——从低负载开始，逐步增加。

### 高端系统：现代 12+ 核（24+ 线程）

CPU 可轻松应对约 250-500M 像素/秒的吞吐量。如有可用硬件编码器，效果更佳。

```bash
# 3摄像头×1280x720x3 @30fps：默认配置即可，可选增大编码器并行度
# 2摄像头×1920x1080x3 @30fps：默认配置即可，可选增大编码器并行度
lerobot-record --dataset.encoder_threads=5 ...

# 3摄像头×1920x1080x3 @30fps：可能需要一些调整
```

### 中端系统：现代 8+ 核（16+ 线程）或 Apple Silicon

CPU 可应对约 80-300M 像素/秒的吞吐量。

```bash
# 3摄像头×640x480x3 @30fps：默认配置即可，可选减小编码器并行度
# 2摄像头×1280x720x3 @30fps：默认配置即可，可选减小编码器并行度
lerobot-record --dataset.encoder_threads=2 ...

# 2摄像头×1920x1080x3 @30fps：可能需要一些调整
```

### 低资源系统：现代 4+ 核（8+ 线程）或 Raspberry Pi 5

在资源极度受限的系统上，流式编码可能与采集循环竞争过于激烈。禁用后将回退到基于 PNG 的方式，编码在片段之间进行（阻塞式，但不干扰采集）。也可以降低录制吞吐量以同时减轻采集和编码负担，或考虑切换为 `h264` 编解码器并使用批量编码。

```bash
# 2摄像头×640x480x3 @30fps：需要一些调整

# 使用 H.264，禁用流式编码，考虑批量编码
lerobot-record --dataset.vcodec=h264 --dataset.streaming_encoding=false ...
```

## 7. 结语

性能最终取决于你的具体配置——帧率、分辨率、CPU 核心数及负载、可用内存、片段时长以及所选编码器。请始终使用目标工作负载进行测试，充分了解你的 CPU 和系统能力，并合理调整 `encoder_threads`、`encoder_queue_maxsize` 和 `vcodec`。话虽如此，一个常见的实用配置（适用于许多应用场景）是三个摄像头以 640×480×3 @30fps 录制；在现代系统上，使用默认流式视频编码设置通常可以正常运行。务必通过对比视频时长与 CLI 片段时长、确认行数等于 FPS × CLI 时长来验证录制数据集的健康状态。
