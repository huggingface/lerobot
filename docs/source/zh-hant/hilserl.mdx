# HIL-SERL 實體機器人訓練工作流程指南（HIL-SERL real robot training workflow guide）

在本教學中，您將使用 LeRobot 完整體驗人在迴圈（human-in-the-loop）中的樣本高效強化學習（Human-in-the-Loop Sample-Efficient Reinforcement Learning，HIL-SERL）工作流程。您將在短短幾小時內，掌握透過強化學習（RL）在實體機器人上訓練一個策略（training a policy）。

HIL-SERL 是一種樣本高效的強化學習（sample-efficient reinforcement learning）演算法，結合了人類示範（human demonstrations）、線上學習（online learning）與人類介入（human interventions）。該方法從少量的人類示範開始，利用這些示範訓練一個獎勵分類器（train a reward classifier），然後採用行動者-學習者（actor-learner）架構，讓人類可以在策略（policy）執行過程中隨時介入（intervene），以引導探索（guide exploration）並修正不安全的行為（unsafe behaviors）。在本教學中，您將使用遊戲手把來提供介入操作並控制機器人的學習過程。

它結合了三個關鍵要素：

1. **離線示範與獎勵分類器（offline demonstrations & reward classifier）：** 少量的人類遙操作回合（human-teleop episodes）加上基於視覺的成功偵測器（vision-based success detector），為策略（policy）提供一個塑造過的起始點（shaped starting point）。

2. **機器人上的行動者/學習者迴圈與人類介入（on-robot actor / learner loop with human interventions）：** 分散式的 SAC（distributed Soft Actor Critic）行動者（actor）探索實體機器人的同時更新策略（policy）；人類可以隨時介入以修正危險或無效的行為（dangerous or unproductive behaviour）。

3. **安全與效率工具（safety & efficiency tools）：** 關節（joint）/末端執行器（end-effector, EE）界限（bounds）、裁切（crop）感興趣區域（ROI）前處理以及 WandB 監控，確保資料有用以及硬體安全。

這些要素結合在一起，使 HIL-SERL 能達到近乎完美的任務成功（task success），且循環時間（cycle times）比純模仿學習的基線（imitation-only baselines）更快。

## 我需要什麼？（What do I need?）

- 一個遊戲手把（建議）或鍵盤來控制機器人
- 一張 Nvidia GPU
- 一個配備從動臂和主動臂（follower and leader arm）的實體機器人（如果使用鍵盤或遊戲手把則為可選）
- 機器人的 URDF 檔案，供運動學套件（kinematics package）使用（請參考 `lerobot/model/kinematics.py`）

## 我可以訓練什麼類型的任務?（What kind of tasks can I train?）

您可以使用 HIL-SERL 訓練（train）各種操作任務（manipulation tasks）。以下是一些建議：

- 從簡單的任務（simple task）開始，以了解系統的運作方式。
  - 將方塊推到目標區域
  - 用夾爪（gripper）夾取並抬起方塊
- 避免極長水平任務（extremely long horizon tasks）。專注於可以在 5-10 秒內完成的任務（tasks）。
- 一旦您對系統的運作方式有了良好的了解，就可以嘗試更複雜的任務（more complex tasks）和更長水平（longer horizons）。
  - 夾取並放置方塊
  - 使用雙臂夾取物體的雙臂任務（bimanual task）
  - 從一隻手臂轉移物體到另一隻手臂的交接任務（hand-over task）
  - 盡情發揮！

## 安裝帶有 HIL-SERL 的 LeRobot（install LeRobot with HIL-SERL）

要安裝帶有 HIL-SERL 的 LeRobot，您需要安裝 `hilserl` 額外套件（extra）。

```bash
pip install -e ".[hilserl]"
```

## 實體機器人訓練工作流程（real robot training workflow）

### 了解配置（understanding configuration）

訓練過程（training process）從 HILSerl 環境的正確配置（proper configuration）開始。主要的配置類（main configuration class）是 `lerobot/rl/gym_manipulator.py` 中的 `GymManipulatorConfig`，其中包含巢狀（nested）的 `HILSerlRobotEnvConfig` 和 `DatasetConfig`。配置（configuration）被組織成清晰的巢狀子配置（nested sub-configs）：

<!-- prettier-ignore-start -->
```python
class GymManipulatorConfig:
    env: HILSerlRobotEnvConfig    # 環境配置（巢狀）
    dataset: DatasetConfig    # 資料集記錄/重播配置（巢狀）
    mode: str | None = None    # "record"、"replay" 或 None（用於訓練）
    device: str = "cpu"    # 運算裝置

class HILSerlRobotEnvConfig(EnvConfig):
    robot: RobotConfig | None = None    # 主要機器人代理（定義於 lerobot/robots）
    teleop: TeleoperatorConfig | None = None    # 遙操作代理，例如遊戲手把或主動臂
    processor: HILSerlProcessorConfig    # 處理管線配置（巢狀）
    name: str = "real_robot"    # 環境名稱
    task: str | None = None    # 任務識別碼
    fps: int = 10    # 控制頻率

# 巢狀處理器配置
class HILSerlProcessorConfig:
    control_mode: str = "gamepad"    # 控制模式
    observation: ObservationConfig | None = None    # 觀測處理設定
    image_preprocessing: ImagePreprocessingConfig | None = None    # 影像裁切/縮放設定
    gripper: GripperConfig | None = None    # 夾爪控制與懲罰設定
    reset: ResetConfig | None = None    # 環境重置與計時設定
    inverse_kinematics: InverseKinematicsConfig | None = None    # 逆運動學處理設定
    reward_classifier: RewardClassifierConfig | None = None    # 獎勵分類器設定
    max_gripper_pos: float | None = 100.0    # 最大夾爪位置

# 子配置類
class ObservationConfig:
    add_joint_velocity_to_observation: bool = False    # 將關節速度加入狀態
    add_current_to_observation: bool = False    # 將馬達電流加入狀態
    add_ee_pose_to_observation: bool = False    # 將末端執行器姿態加入狀態
    display_cameras: bool = False    # 執行期間顯示相機畫面

class ImagePreprocessingConfig:
    crop_params_dict: dict[str, tuple[int, int, int, int]] | None = None    # 影像裁切參數
    resize_size: tuple[int, int] | None = None    # 目標影像大小

class GripperConfig:
    use_gripper: bool = True    # 啟用夾爪控制
    gripper_penalty: float = 0.0    # 不當使用夾爪的懲罰

class ResetConfig:
    fixed_reset_joint_positions: Any | None = None    # 重置時的關節位置
    reset_time_s: float = 5.0    # 重置期間的等待時間
    control_time_s: float = 20.0    # 最大回合持續時間
    terminate_on_success: bool = True    # 偵測到成功時是否終止回合

class InverseKinematicsConfig:
    urdf_path: str | None = None # 機器人 URDF 檔案路徑
    target_frame_name: str | None = None # 末端執行器座標系名稱
    end_effector_bounds: dict[str, list[float]] | None = None # 末端執行器工作空間界限
    end_effector_step_sizes: dict[str, float] | None = None # 各軸的末端執行器步進大小

class RewardClassifierConfig:
    pretrained_path: str | None = None    # 預訓練獎勵分類器路徑
    success_threshold: float = 0.5    # 成功偵測閾值
    success_reward: float = 1.0    # 成功回合的獎勵值

# 資料集配置
class DatasetConfig:
    repo_id: str    # LeRobot 資料集儲存庫 ID
    task: str    # 任務識別碼
    root: str | None = None    # 本地資料集根目錄
    num_episodes_to_record: int = 5    # 錄製的回合數
    replay_episode: int | None = None    # 重播的回合索引
    push_to_hub: bool = False    # 是否將資料集推送到 Hub
```

### 處理器管線架構（processor pipeline architecture）

HIL-SERL 使用模組化的處理器管線架構（modular processor pipeline architecture），透過一系列可組合的步驟（composable steps）來處理機器人觀測（observations）和動作（actions）。管線（pipeline）分為兩個主要組件（two main components）：

#### 環境處理器管線（environment processor pipeline）

環境處理器（environment processor, `env_processor`）處理傳入的觀測（observations）和環境狀態（environment state）：

1. **VanillaObservationProcessorStep**：將原始機器人觀測（raw robot observations）轉換為標準化格式（standardized format）
2. **JointVelocityProcessorStep**（可選）：將關節速度（joint velocity）資訊加入觀測（observations）
3. **MotorCurrentProcessorStep**（可選）：將馬達電流（motor current）讀數加入觀測（observations）
4. **ForwardKinematicsJointsToEE**（可選）：從關節（joint）位置計算末端執行器（end-effector）姿態
5. **ImageCropResizeProcessorStep**（可選）：裁切（crops）和調整攝影機影像大小（resizes）
6. **TimeLimitProcessorStep**（可選）：強制執行回合時間限制（episode time limits）
7. **GripperPenaltyProcessorStep**（可選）：對不當使用夾爪（inappropriate gripper usage）施加懲罰（penalties）
8. **RewardClassifierProcessorStep**（可選）：使用視覺模型（vision models）自動偵測獎勵（automated reward detection）
9. **AddBatchDimensionProcessorStep**：將資料轉換為批次格式（batch format）以供神經網路處理（neural network processing）
10. **DeviceProcessorStep**：將資料移至指定的計算裝置（CPU/GPU）

#### 動作處理器管線（action processor pipeline）

動作處理器（action processor, `action_processor`）處理輸出的動作（outgoing actions）和人類介入（human interventions）：

1. **AddTeleopActionAsComplimentaryDataStep**：擷取遙操作動作（teleoperator action）以供記錄（logging）
2. **AddTeleopEventsAsInfoStep**：記錄介入事件（records intervention events）和回合控制訊號（episode control signals）
3. **InterventionActionProcessorStep**：處理人類介入（human interventions）和回合終止（episode termination）
4. **逆運動學管線（inverse kinematics pipeline）**（啟用時）：
   - **MapDeltaActionToRobotActionStep**：將差量動作（delta actions）轉換為機器人動作格式（robot action format）
   - **EEReferenceAndDelta**：計算末端執行器參考（end-effector reference）和差量移動（delta movements）
   - **EEBoundsAndSafety**：強制執行工作空間安全邊界（workspace safety bounds）
   - **InverseKinematicsEEToJoints**：將末端執行器動作（end-effector actions）轉換為關節目標（joint targets）
   - **GripperVelocityToJoint**：處理夾爪（gripper）控制命令

### 進階的觀測處理（advanced observation processing）

HIL-SERL 框架支援額外的觀測處理（observation processing）功能，可以改善策略的學習（policy learning）：

#### 關節速度處理（joint velocity processing）

啟用關節速度估計（joint velocity estimation），為策略（policy）提供運動資訊（motion information）：

```json
{
  "env": {
    "processor": {
      "observation": {
        "add_joint_velocity_to_observation": true
      }
    }
  }
}
```

此處理器（processor）：

- 使用連續關節位置（consecutive joint position）讀數之間的有限差分（finite differences）來估計關節速度（joint velocities）
- 將速度資訊（velocity information）加入觀測狀態向量（observation state vector）
- 對於需要運動感知（motion awareness）的動態任務（dynamic tasks）的策略（policies）很有用

#### 馬達電流處理（motor current processing）

監控馬達電流（motor currents）以偵測接觸力（contact forces）和負載條件（load conditions）：

```json
{
  "env": {
    "processor": {
      "observation": {
        "add_current_to_observation": true
      }
    }
  }
}
```

此處理器（processor）：

- 從機器人的控制系統讀取馬達電流值（motor current values）
- 將電流量測（current measurements）加入觀測狀態向量（observation state vector）
- 有助於偵測接觸事件（contact events）、物體重量（object weights）和機械阻力（mechanical resistance）
- 對於接觸充分的操作任務（contact-rich manipulation tasks）很有用

#### 組合觀測處理（combined observation processing）

您可以同時啟用多個觀測處理功能（observation processing features）：

```json
{
  "env": {
    "processor": {
      "observation": {
        "add_joint_velocity_to_observation": true,
        "add_current_to_observation": true,
        "display_cameras": false
      }
    }
  }
}
```

**注意**：啟用額外的觀測功能（observation features）會增加狀態空間的維度（state space dimensionality），這可能需要調整您的策略網路（policy network）架構，並可能需要收集更多訓練資料（training data）。

### 尋找機器人工作空間邊界（finding robot workspace bounds）

在收集示範（collecting demonstrations）之前，您需要確定機器人的適當操作邊界（operational bounds）。

這有助於透過兩種方式簡化在實體機器人上學習的問題（problem of learning）：1）將機器人的操作空間（operational space）限制在能解決任務的特定區域（specific region that solves the task），避免不必要或不安全的探索（unnecessary or unsafe exploration）；2）允許在末端執行器空間（end-effector space）而非關節空間（joint space）中進行訓練（training）。根據經驗，對於強化學習（reinforcement learning, RL）中的操作任務，在關節空間中學習通常是更困難的問題——有些任務在關節空間中幾乎不可能學習，但當動作空間（action space）轉換為末端效應器座標（end-effector coordinates）時就變得可學習。

**使用 lerobot-find-joint-limits**

此腳本（script）幫助您找到機器人末端執行器（end-effector）的安全操作邊界（safe operational bounds）。假設您有從臂和主臂（follower and leader arm），您可以使用此腳本找到從臂的邊界（bounds），這些邊界將在訓練（training）期間應用。
限定動作空間（action space）將減少代理的冗餘探索（redundant exploration）並保證安全。

```bash
lerobot-find-joint-limits \
  --robot.type=so100_follower \
  --robot.port=/dev/tty.usbmodem58760431541 \
  --robot.id=black \
  --teleop.type=so100_leader \
  --teleop.port=/dev/tty.usbmodem58760431551 \
  --teleop.id=blue
```

**工作流程（workflow）**

1. 執行腳本（script）並在要解決任務的空間（space that solves the task）中移動機器人
2. 腳本（script）會記錄末端執行器位置（end-effector positions）的最小值和最大值（minimum and maximum）以及關節角度（joint angles），並將它們列印到控制台（console），例如：
   ```
   Max ee position [0.2417 0.2012 0.1027]
   Min ee position [0.1663 -0.0823 0.0336]
   Max joint positions [-20.0, -20.0, -20.0, -20.0, -20.0, -20.0]
   Min joint positions [50.0, 50.0, 50.0, 50.0, 50.0, 50.0]
   ```
3. 將這些值用於環境配置（environment configuration）中的 `env.processor.inverse_kinematics.end_effector_bounds`（參見 `lerobot/envs/configs.py` 中的 `InverseKinematicsConfig`）

**配置範例（example configuration）**

```json
{
  "env": {
    "processor": {
      "inverse_kinematics": {
        "end_effector_bounds": {
          "max": [0.24, 0.2, 0.1],
          "min": [0.16, -0.08, 0.03]
        }
      }
    }
  }
}
```

### 收集示範（collecting demonstrations）

定義好邊界（bounds）後，您可以安全地收集訓練示範（demonstrations for training）。使用 off-policy 演算法進行強化學習（RL） 訓練允許我們使用離線收集的資料集（offline datasets）來提高學習的效率。

**設定錄製模式（setting up record mode）**

建立一個用於錄製示範（recording demonstrations）的配置檔案（configuration file，或編輯現有的檔案，如 [env_config.json](https://huggingface.co/datasets/lerobot/config_examples/resolve/main/rl/env_config.json)）：

1. 在根層（root level）級將 `mode` 設為 `"record"`
2. 在 `dataset` 區段（section）中為您的資料集（dataset）指定唯一的 `repo_id`（例如 "username/task_name"）
3. 在 `dataset` 區段（section）中將 `num_episodes_to_record` 設為您想收集的示範數量（number of demonstrations）
4. 先將 `env.processor.image_preprocessing.crop_params_dict` 初始化為 `{}`（我們稍後會決定裁切 crops）
5. 在 `env` 區段中配置 `env.robot`、`env.teleop` 和其他硬體設定

配置範例片段（example configuration section）：

```json
{
  "env": {
    "type": "gym_manipulator",
    "name": "real_robot",
    "fps": 10,
    "processor": {
      "control_mode": "gamepad",
      "observation": {
        "display_cameras": false
      },
      "image_preprocessing": {
        "crop_params_dict": {},
        "resize_size": [128, 128]
      },
      "gripper": {
        "use_gripper": true,
        "gripper_penalty": 0.0
      },
      "reset": {
        "reset_time_s": 5.0,
        "control_time_s": 20.0
      }
    },
    "robot": {
      // ... 機器人配置 ...
    },
    "teleop": {
      // ... 遙操作器配置 ...
    }
  },
  "dataset": {
    "repo_id": "username/pick_lift_cube",
    "root": null,
    "task": "pick_and_lift",
    "num_episodes_to_record": 15,
    "replay_episode": 0,
    "push_to_hub": true
  },
  "mode": "record",
  "device": "cpu"
}
```

### 使用遙操作裝置（using a teleoperation device）

除了機器人之外，您還需要一個遙操作裝置（teleoperation device）來控制它，以便收集任務（task）的資料集（dataset）並在線上訓練（training）期間進行介入（interventions）。
我們支援使用遊戲手把、鍵盤或機器人的主臂（leader arm）。

HIL-Serl 在機器人的末端執行器空間（end-effector space）中學習動作（action）。因此，遙操作（teleoperation）將控制末端執行器的 x、y、z 位移（end-effector’s x,y,z displacements）。

末端執行器變換（end-effector transformation）由處理器管線（processor pipeline）套用（包含 `InverseKinematicsRLStep`、`EEBoundsAndSafety`、`EEReferenceAndDelta`、`GripperVelocityToJoint`），並配置於 `env.processor.inverse_kinematics`（`InverseKinematicsConfig`）以及 `env.processor.gripper` / `env.processor.max_gripper_pos` 之下。與末端執行器空間（end-effector space）相關的預設值（default）如下：

<!-- prettier-ignore-start -->
```python
class InverseKinematicsConfig:
```

錄製（recording）期間：

1. 機器人將重置（reset）到配置檔案（configuration file） `env.processor.reset.fixed_reset_joint_positions` 中定義的初始位置（initial position）
2. 成功完成任務（task）

### 處理資料集（processing the dataset）

收集示範（collecting demonstrations）後，處理它們以確定最佳的攝影機裁切（optimal camera crops）。
強化學習（reinforcement learning, RL）對背景干擾（background distractions）很敏感，因此將影像裁切（crop）到相關的工作空間區域（relevant workspace area）非常重要。

基於視覺的 RL 演算法直接從像素輸入（pixel inputs）學習，使其容易受到無關視覺資訊（irrelevant visual information）的影響。背景元素（background elements）如變化的光線、陰影、移動的人或工作空間外的物體都可能干擾學習過程（learning process）。好的 ROI 選擇應該：

- 僅包含任務（task）發生的必要工作空間（essential workspace）
- 捕捉機器人的末端效應器（end-effector）和任務（task）涉及的所有物體（objects）
- 排除不必要的背景元素（background elements）和干擾（distractions）

注意：如果您已經知道裁切參數（crop parameter），可以跳過此步驟，直接在錄製（recording）期間的配置檔案（onfiguration file）中設定 `crop_params_dict`。

**決定裁切參數（determining crop parameters）**

使用 `crop_dataset_roi.py` 腳本互動式地選擇攝影機影像（camera images）中的感興趣區域（regions of interest）：


1. 對於每個攝影機視角（camera view），腳本將顯示第一影格（first frame）
2. 在相關的工作空間區域（elevant workspace area）周圍繪製矩形
3. 按 'c' 確認選擇
4. 對所有攝影機視角（camera views）重複操作
5. 腳本輸出裁切參數（crop parameter）並建立新的裁切資料集（dataset）

### 訓練獎勵分類器（training a reward classifier）

獎勵分類器（reward classifier）在 HIL-SERL 工作流程中扮演重要角色，透過自動化獎勵分配（automating reward assignment）和自動偵測回合成功（automatically detecting episode success）。與其手動定義獎勵函數（reward functions）或每個時間步（timestep）都依賴人類回饋（human feedback），獎勵分類器（reward classifier）學習從視覺觀測中預測成功（success）/失敗（failure）。這使得強化學習（RL）演算法能夠根據機器人攝影機輸入（camera inputs）提供一致且自動化的獎勵訊號（consistent and automated reward signals），從而高效學習。

本指南說明如何訓練（training） LeRobot 為人在回圈中強化學習（human-in-the-loop reinforcement learning, RL）實作的獎勵分類器（reward classifier）。獎勵分類器學習在給定狀態（state）下預測獎勵值（predict the reward value），可用於強化學習（RL）設定中訓練策略（train a policy）。

**注意**：訓練獎勵分類器（training a reward classifier）是可選的。您可以透過使用遊戲手把或鍵盤裝置手動標註成功（annotating the success）來開始第一輪強化學習（RL）實驗。

`modeling_classifier.py` 中的獎勵分類器（reward classifier）實作使用預訓練的視覺模型（pretrained vision model）來處理影像（process the images）。它可以輸出單一值用於二元獎勵（binary rewards）以預測成功（success）/失敗（failure）情況，或輸出多個值用於多類別（multi-class）設定。

**為獎勵分類器收集資料集（collecting a dataset for the reward classifier）**

在訓練（training）之前，您需要收集帶有標註範例（labeled examples）的資料集（dataset）。在配置中設定 `mode: "record"` 並執行 `gym_manipulator.py`，即可啟動收集觀測（observations）、動作（actions）和獎勵（rewards）資料集的流程。

要收集資料集（dataset），您需要根據 HILSerlRobotEnvConfig 修改環境配置中的一些參數（parameters）。

```bash
python -m lerobot.rl.gym_manipulator --config_path src/lerobot/configs/reward_classifier_train_config.json
```

**資料收集的關鍵參數（key parameters for data collection）**

- **mode**：設為 `"record"` 以收集資料集（collect a dataset，在根層級 root level）
- **dataset.repo_id**：`"hf_username/dataset_name"`，資料集（dataset）名稱和 Hub 上的儲存庫（repo）名稱
- **dataset.num_episodes_to_record**：要錄製（record）的回合（episode）數
- **env.processor.reset.terminate_on_success**：是否在偵測到成功（success）時自動終止回合（automatically terminate episodes）（預設：`true`）
- **env.fps**：每秒錄製的影格數（number of frames per second to record）
- **dataset.push_to_hub**：是否將資料集（dataset）推送到 Hub

`env.processor.reset.terminate_on_success` 參數（parameter）允許您控制回合終止行為（episode termination behavior）。設為 `false` 時，即使偵測到成功，回合（episodes）也會繼續，允許您收集更多帶有 reward=1 標籤的正面範例（positive examples）。這對於訓練獎勵分類器（training reward classifiers）至關重要，因為它在您的資料集（dataset）中提供更多成功狀態的範例（success state examples）。設為 `true`（預設）時，回合會在偵測到成功（success detection）時立即終止。

**重要**：對於獎勵分類器訓練（reward classifier training），將 `terminate_on_success` 設為 `false` 以收集足夠的正面範例（sufficient positive examples）。對於一般的 HIL-SERL 訓練（training），保持為 `true` 以在任務（task）成功完成時啟用自動終止回合（episode）。

### 使用執行者-學習者架構進行訓練（training with actor-learner）

LeRobot 系統使用分散式執行者-學習者（distributed actor-learner）架構進行訓練（training）。此架構將機器人互動與學習過程解耦（decouples），使它們能夠並行執行而不互相阻塞（run concurrently without blocking eachother）。執行者伺服器（actor server）處理機器人觀測（observations）和動作（actions），將互動資料（interaction data）發送給學習者伺服器（learner server）。學習者伺服器（learner server）執行梯度下降（gradient descent）並定期更新（periodically updates）執行者的策略權重（actor’s policy weights）。您需要啟動兩個處理程序（processes）：一個學習者（learner）和一個執行者（actor）。

**配置設定（configuration setup）**

建立訓練（training）配置檔案（training configuration file，範例可在[此處](https://huggingface.co/datasets/lerobot/config_examples/resolve/main/rl/train_config.json)取得）。訓練配置（training config）基於 `lerobot/rl/train_rl.py` 中的主要 `TrainRLServerPipelineConfig` 類（class）。

1. 配置策略設定（policy settings，`type="gaussian_actor"`、`device` 等）
2. 在頂層的 `algorithm` 區塊中配置演算法設定（algorithm settings，`type="sac"`、學習率 learning rates、折扣係數 discont 等，定義於 `lerobot/rl/algorithms/sac/configuration_sac.py`）。
3. 將 `dataset` 設為您裁切過的資料集（cropped dataset）
4. 配置環境設定（cropped dataset）和裁切參數（crop parameters）
5. 檢查 [configuration_gaussian_actor.py](https://github.com/huggingface/lerobot/blob/main/src/lerobot/policies/gaussian_actor/configuration_gaussian_actor.py#L79) 中與高斯執行者（Gaussian Actor）相關的其他參數（parameters）。
6. 驗證 `policy` 配置中的 `input_features` 和 `output_features` 對於您的任務（task）是正確的。

**啟動學習者（starting the learner）**

首先，啟動學習者伺服器處理程序（learner server process）：

```bash
python -m lerobot.rl.learner --config_path src/lerobot/configs/train_config_hilserl_so100.json
```

學習者（learner）：

- 初始化策略的網路（policy network）
- 準備經驗重播緩衝區（replay buffers）
- 開啟 `gRPC` 伺服器（server）並與執行者通訊（communicate with actors）
- 處理策略（policy）的轉移與更新（transitions and updates）

**啟動執行者（starting the actor）**

在另一個終端機（terminal）中，使用相同的配置（configuration）啟動執行者處理程序（actor process）：

```bash
python -m lerobot.rl.actor --config_path src/lerobot/configs/train_config_hilserl_so100.json
```

執行者（actor）：

- 透過 `gRPC` 連接到學習者（learner）
- 初始化環境（environment）
- 執行策略（policy）的 rollouts 以收集經驗（collect experience）
- 將轉移（transitions）發送給學習者（learner）
- 接收更新後的策略參數（updated policy parameters）

**訓練流程（training flow）**

訓練（training）會自動進行：

1. 執行者（actor）在環境（environment）中執行策略（policy）
2. 收集轉移（transitions）並發送給學習者（learner）
3. 學習者（learner）根據這些轉移（transitions）更新策略（policy）
4. 更新後的策略參數（updated policy parameters）發送回執行者（actor）
5. 過程持續進行直到達到指定的步數限制（specified step limit）

**人在迴圈（human-in-the-loop）**

- 高效學習的關鍵是透過人類介入（human interventions）提供修正回饋（corrective feedback）並完成任務（task），以協助策略（policy）學習和探索（learning and exploration）。
- 要進行人類介入（human intervention），您可以按遊戲手把上的右上方扳機按鈕（或鍵盤上的 `space` 鍵）。這將暫停策略動作（policy actions），允許您接管控制（take over）。
- 成功的實驗是在開始時人類需要頻繁介入（intervene），但隨著策略（policy）改善逐漸減少介入次數。您可以在 `wandb` 儀表板中監控介入率（intervention rate）。

### 人為介入指南（guide to human interventions）

學習過程對介入策略（intervention strategy）非常敏感。需要幾次嘗試才能理解如何有效地介入（intervene effectivel）。一些提示和建議：

- 在訓練（training）開始時允許策略（policy）自行探索（explore）幾個回合（episodes）。
- 避免長時間介入（intervening）。盡量在機器人行為偏離軌道（off track）時進行介入以糾正行為。
- 一旦策略（policy）開始完成任務（task），即使不完美，您可以將介入限制在簡單的快速動作（actions），例如簡單的抓取命令。

### 需要調整的關鍵超參數（key hyperparameters to tune）

某些配置值（configuration values）對訓練（training）穩定性和速度具有不成比例的影響：

- **`temperature_init`**（`algorithm.temperature_init`）——SAC 中的初始熵溫度（initial entropy temperature）。較高的值鼓勵更多探索（exploration）；較低的值使策略（policy）在早期更具確定性（more deterministic）。一個好的起始點是 `1e-2`。我們觀察到設定過高會使人為介入（human interventions）無效並減慢學習速度。
- **`policy_parameters_push_frequency`**（`policy.actor_learner_config.policy_parameters_push_frequency`）——從學習者（learner）向執行者（actor）推送兩次權重（weight）之間的間隔（以*秒*為單位）。預設值為 `4 s`。減少到 **1-2 s** 以提供更新鮮的權重（但會增加更多網路流量 network traffic）；僅在連線速度慢時才增加此值，因為這會降低樣本效率（sample efficiency）。
- **`storage_device`**（`policy.storage_device`）——學習者（learner）保存策略參數（policy parameters）的裝置。如果您有多餘的 GPU 記憶體，將其設為 `"cuda"`（而非預設的 `"cpu"`）。將權重（weights）保留在 GPU 上可以消除 CPU→GPU 傳輸開銷（transfer overhead），並能顯著增加每秒學習者更新的次數。
