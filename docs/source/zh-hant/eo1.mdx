# EO-1

EO-1 是一個**用於機器人控制的視覺-語言-動作策略（Vision-Language-Action policy for robot control）**。LeRobot 的實作將 EO-1 整合進標準的 LeRobot 訓練（training）、評估（evaluation）與處理器介面（processor interface）。

## 模型概覽（model overview）

EO-1 使用 Qwen2.5-VL 骨幹網路（backbone）進行視覺與語言理解（vision-language understanding），並加入一個連續的流匹配動作頭（continuous flow-matching action head）來控制機器人。此策略（policy）將每個機器人控制樣本（robot-control sample ）格式化為一段多模態對話（multimodal conversation）：攝影機影像（camera images）會傳入 Qwen2.5-VL，機器人狀態（robot state）以 EO-1 狀態 token （state tokens）表示，而未來的動作區塊（future action chunk）則以 EO-1 動作 token （action tokens）表示。

在訓練（training）期間，EO-1 學習在動作 token 的位置（action-token positions）對連續動作區塊（continuous action chunks）進行去噪（denoise）。在推論（inference）期間，它會取樣一個動作區塊（samples an action chunk）、回傳連續動作（returns continuous actions），並在再次取樣（sampling）前先執行該區塊（chunk）中的 `n_action_steps` 個動作。

### LeRobot 整合涵蓋的範圍（what the lerobot integration covers）

- 透過 LeRobot 的標準 `policy.type=eo1` 設定（configuration）
- 透過策略處理器（policy processors）進行 Qwen2.5-VL 影像（image）與文字預處理（text preprocessing）
- 連續流匹配動作預測（continuous flow-matching action prediction）
- 透過 LeRobot 策略（policy） API 進行檢查點（checkpoint）的儲存與載入
- 使用 `lerobot-train` 訓練（training），並以 `lerobot-eval` 評估（evaluation）

更廣義的 EO-1 專案也包含交錯式的視覺-文字-動作預訓練（interleaved vision-text-action pretraining），以及多模態推理工作流程（multimodal reasoning workflows）。本頁聚焦於 LeRobot 的機器人控制策略路徑（robot-control policy path）。

## 安裝需求（installation requirements）

1. 依照[安裝指南（Installation Guide）](./installation)安裝 LeRobot。
2. 執行以下指令安裝 EO-1 相依套件（dependencies）：

   ```bash
   pip install -e ".[eo1]"
   ```

3. 如果您想在 LIBERO 上訓練（training）或評估，也請安裝 LIBERO 相依套件（dependencies）：

   ```bash
   pip install -e ".[eo1,libero]"
   ```

EO-1 可以透過 `policy.attn_implementation=sdpa` 使用標準的 PyTorch scaled-dot-product attention 後端（backend）。如果您的環境已安裝相容的 `flash_attn`，可以指定 `policy.attn_implementation=flash_attention_2`。

## 資料需求（data requirements）

EO-1 需要一個包含以下內容的 LeRobot 資料集（dataset）：

- 至少一個視覺觀測（visual observation），例如 `observation.images.image`
- `observation.state`
- `action`
- 透過資料集（dataset） `task` 欄位（field）提供的語言任務指令（language task instruction）

如果您的資料集（dataset）使用不同的觀測名稱（observation names），請使用 `rename_map` 將它們對齊到您的訓練或評估設定（training or evaluation setup）所預期的名稱（names）。

## 使用方式（usage）

若要在 LeRobot 設定中使用 EO-1，請將策略類型（policy type）指定為：

```python
policy.type=eo1
```

預設情況下，新的 EO-1 策略（policy）會從以下來源初始化其骨幹網路（backbone）：

```python
policy.vlm_base=Qwen/Qwen2.5-VL-3B-Instruct
```

當您有可用的 LeRobot 格式 EO-1 檢查點（checkpoint）後，以下列方式載入：

```python
policy.path=your-org/your-eo1-checkpoint
```

## 訓練（training）

### 訓練指令範例（training command example）

```bash
lerobot-train \
  --dataset.repo_id=your_org/your_dataset \
  --policy.type=eo1 \
  --policy.vlm_base=Qwen/Qwen2.5-VL-3B-Instruct \
  --policy.dtype=bfloat16 \
  --policy.attn_implementation=sdpa \
  --policy.gradient_checkpointing=false \
  --output_dir=./outputs/eo1_training \
  --job_name=eo1_training \
  --steps=300000 \
  --batch_size=16 \
  --policy.device=cuda
```

### 主要訓練參數（key training parameters）

| 參數（parameter）                      | 預設值（default）             | 描述（description）                                                                             |
| -------------------------------------- | ----------------------------- | ----------------------------------------------------------------------------------------------- |
| `policy.vlm_base`                      | `Qwen/Qwen2.5-VL-3B-Instruct` | 用於初始化新策略（initialize a new policy）的 Qwen2.5-VL 檢查點（checkpoint）                   |
| `policy.dtype`                         | `auto`                        | 骨幹網路（backbone） dtype 設定：`auto`、`bfloat16` 或 `float32`                                |
| `policy.attn_implementation`           | `None`                        | 選用的 Qwen attention 後端（backend），例如 `sdpa`                                              |
| `policy.gradient_checkpointing`        | `false`                       | 降低訓練（training）期間的記憶體用量                                                            |
| `policy.chunk_size`                    | `8`                           | 每個區塊預測的未來動作數量（number of future actions predicted per chunk）                      |
| `policy.n_action_steps`                | `8`                           | 從取樣區塊中實際採用的動作數量（number of actions consumed from a sampled chunk）               |
| `policy.num_denoise_steps`             | `10`                          | 取樣期間使用的流匹配去噪步數（number of flow-matching denoising steps used during sampling）    |
| `policy.max_state_dim`                 | `32`                          | 狀態填充維度（state padding dimension）                                                         |
| `policy.max_action_dim`                | `32`                          | 動作填充維度（action padding dimension）                                                        |
| `policy.force_fp32_autocast`           | `true`                        | 即使骨幹網路（backbone）使用混合精度（mixed precision），仍將 flow 頭（flow head）保持在 fp32   |
| `policy.supervise_padding_action_dims` | `true`                        | 控制是否對填充的動作維度進行監督（controls whether padded action dimensions are supervised）    |
| `policy.supervise_padding_actions`     | `true`                        | 控制是否對填充的未來動作列進行監督（controls whether padded future action rows are supervised） |

## 評估（evaluation）

有了 LeRobot 格式的檢查點（checkpoint）後，即可透過 `lerobot-eval` 評估（evaluate） EO-1：

```bash
lerobot-eval \
  --policy.path=your-org/your-eo1-checkpoint \
  --env.type=libero \
  --env.task=libero_object \
  --eval.batch_size=1 \
  --eval.n_episodes=20
```

若資料集（dataset）或環境（environments）的攝影機名稱（camera names）與檢查點設定（checkpoint configuration）不同，請傳入 `rename_map`：

```bash
lerobot-eval \
  --policy.path=your-org/your-eo1-checkpoint \
  --env.type=libero \
  --env.task=libero_object \
  --rename_map='{"observation.images.image2":"observation.images.wrist_image"}'
```

## 設定注意事項（configuration notes）

### 影像處理（image processing）

EO-1 使用 Qwen2.5-VL 處理器（processor）。`policy.image_min_pixels` 與 `policy.image_max_pixels` 設定會控制在視覺 token （visual tokens）傳入骨幹網路（backbone）前的影像縮放範圍（image resizing bounds）。

### 狀態與動作維度（state and action dimensions）

此策略（policy）會在進入 EO-1 flow 頭（flow head）之前，將狀態與動作向量（state and action vectors）填充（pads）至 `policy.max_state_dim` 與 `policy.max_action_dim`。策略（policy）回傳預測結果前，會將其裁剪回（cropped back）原本的動作維度（original action dimension）。

### Attention 後端（attention Backend）

若要可攜性佳的設定（portable setup），請使用 `policy.attn_implementation=sdpa`。只有在 `flash_attn` 已安裝且與您的環境相容時，才使用 `flash_attention_2`。
