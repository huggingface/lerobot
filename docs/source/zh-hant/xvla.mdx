# X-VLA：首個適用於任何機器人、任何任務的軟提示機器人基礎模型（the first soft-prompted robot foundation model for any robot, any task）

## 概覽（overview）

多年來，機器人學一直致力於構建能夠遵循自然人類指令（natural human instructions）、並在多種環境（environments）和機器人本體（robot bodies）上進行靈巧操作（operate dexterously）的代理（agent）。近期大型語言模型（LLMs）和視覺語言模型（VLMs）的突破性進展指出了一條前進道路：透過將這些基礎模型架構（foundation-model architectures）扎根於動作（grounding them in actions），將其擴展至具身控制（embodied control）。這催生了視覺-語言-動作（Vision-Language-Action, VLA）模型的興起，期望單一的通用模型（generalist model）能夠結合廣泛的語義理解（semantic understanding ）與穩健的操控技能（robust manipulation skills）。

然而，訓練（training）此類模型（model）是困難的。機器人資料（robot data）分散在不同的平台（platforms）、感測器（sensors）、具身形態（embodiments）和資料收集協議（collection protocols）之間。異質性（heterogeneity）無處不在：不同的機械臂配置（arm configurations）、不同的動作空間（action spaces）、不同的攝影機設置（camera setups）、不同的視覺域（visual domains），以及不同的任務分佈（task distributions）。這些不一致性（inconsistencies）造成了嚴重的分佈偏移（major distribution shifts），使得預訓練不穩定（pretraining unstable）、適配不可靠（adaptation unreliable）。

受元學習（meta-learning）和提示學習（prompt learning）的啟發，我們提出一個問題：**「如果 VLA 模型能夠像 LLM 透過提示學習任務一樣，學習每個機器人和資料集的結構，會怎麼樣？」**

**X-VLA** 是一個軟提示（soft-prompted）、流匹配（flow-matching）的 VLA 框架（framework），它將每種硬體設置視為一個「任務（task）」，並使用一個小集合的可學習的嵌入（learnable embeddings）對其進行編碼（encodes）。這些**軟提示（Soft Prompts）**捕捉了具身形態（embodiment）和領域特定的變異（domain-specific variations），從多模態融合（multimodal fusion）的最早階段（earliest stages）就引導（guiding）Transformer。藉由這一機制，X-VLA 能夠在單一統一架構（unified architecture）中調和不同的機器人形態（robot morphologies）、資料類型（data types）和感測器配置（sensor setups）。

X-VLA 表明，通用機器人智慧（generalist robot intelligence）不需要越來越複雜的架構（architectures），只需要正確的方式來吸收異質性（absorb heterogeneity）。軟提示（Soft Prompts）提供了一種簡單、可擴展的機制（simple, scalable mechanism），用於統一多樣化的機器人資料（unifying diverse robotic data），為可適配的跨具身形態機器人基礎模型（adaptable, cross-embodiment robot foundation models）鋪平了道路。

## 安裝（installation）

安裝 LeRobot 後，安裝 X-VLA 的依賴項（dependencies）：

```bash
pip install -e .[xvla]
```

新版本發布後，您可以這樣安裝：

```bash
pip install lerobot[xvla]
```

## 快速開始（quick start）

### 基本用法（basic usage）

要在 LeRobot 配置中使用 X-VLA，請將策略類型（policy type）指定為：

```bash
policy.type=xvla
```

### 評估預訓練檢查點（evaluating pre-trained checkpoints）

以 LIBERO 為例的評估（evaluation）：

```bash
lerobot-eval \
  --policy.path="lerobot/xvla-libero" \
  --env.type=libero \
  --env.task=libero_spatial,libero_goal,libero_10 \
  --env.control_mode=absolute \
  --eval.batch_size=1 \
  --eval.n_episodes=1 \
  --env.episode_length=800 \
  --seed=142
```

## 可用的檢查點（available checkpoints）

### 🎯 基礎模型（base model）

**[lerobot/xvla-base](https://huggingface.co/lerobot/xvla-base)**

X-VLA 的 0.9B 參數實例（parameter instantiation），使用精心設計的資料處理（data processing）和學習方案（learning recipe）進行訓練（training）。訓練管線（training pipeline）包含兩個階段（phases）：

- **第一階段（Phase I）：預訓練（pretrained）** - 在來自 Droid、Robomind 和 Agibot 的 290K 個回合（episodes）上進行預訓練（pretraining），涵蓋七個平台（platforms）、五種類型的機械臂（從單臂 single-arm 到雙臂 bi-manual 設置）。透過利用軟提示（soft prompts）吸收具身形態特定的變異（absorb embodiment-specific variations），模型（model）學習了一個與具身形態無關的通用策略（embodiment-agnostic generalist policy）。

- **第二階段（Phase II）：領域適配（domain adaptation）** - 針對目標領域（target domains）適配為可部署的策略（deployable policies）。引入一組新的軟提示（soft prompts）並進行最佳化，以編碼（encode）新領域（novel domain）的硬體配置（hardware configuration），同時保持預訓練主幹網路（pretrained backbone）凍結（frozen）。

## 訓練 X-VLA（training X-VLA）

### 建議的訓練配置（recommended training configuration）

在為新的具身形態或任務（embodiment or task）微調（fine-tuning） X-VLA 時，我們建議不要凍結（freezing）VLM，並設置 `policy.dtype=bfloat16` 以避免記憶體不足錯誤（OOM error）。

```bash
lerobot-train \
  --dataset.repo_id=YOUR_DATASET \
  --output_dir=./outputs/xvla_training \
  --job_name=xvla_training \
  --policy.path="lerobot/xvla-base" \
  --policy.repo_id="HF_USER/xvla-your-robot" \
  --policy.dtype=bfloat16 \
  --policy.action_mode=auto \
  --steps=20000 \
  --policy.device=cuda \
  --policy.freeze_vision_encoder=false \
  --policy.freeze_language_encoder=false \
  --policy.train_policy_transformer=true \
  --policy.train_soft_prompts=true \
```

### 訓練參數說明（training parameters explained）

| 參數（parameter）          | 預設值（default） | 描述（description）                                                         |
| -------------------------- | ----------------- | --------------------------------------------------------------------------- |
| `freeze_vision_encoder`    | `false`           | 不凍結 VLM 視覺編碼器權重（do not freeze the VLM vision encoder weights）   |
| `freeze_language_encoder`  | `false`           | 不凍結 VLM 語言編碼器權重（do not freeze the VLM language encoder weights） |
| `train_policy_transformer` | `true`            | 允許策略 Transformer 層參與訓練（allow policy transformer layers to train） |
| `train_soft_prompts`       | `true`            | 允許軟提示參與訓練（allow soft prompts to train）                           |

**💡 最佳實踐**：對於第二階段（Phase II）適配新的具身形態（embodiments），不要凍結（freeze）VLM 編碼器（encoder），同時訓練策略（train the policy）Transformer 和軟提示（soft prompts）。

### 範例：在雙臂機器人上訓練（training on bimanual robot）

```bash
lerobot-train \
  --dataset.repo_id=<USER>/bimanual-so100-handover-cube \
  --output_dir=./outputs/xvla_bimanual \
  --job_name=xvla_so101_training \
  --policy.path="lerobot/xvla-base" \
  --policy.dtype=bfloat16 \
  --policy.repo_id="YOUR_USERNAME/xvla-biso101" \
  --steps=3000 \
  --policy.device=cuda \
  --policy.action_mode=so101_bimanual \
  --policy.freeze_vision_encoder=false \
  --policy.freeze_language_encoder=false \
  --policy.train_policy_transformer=true \
  --policy.train_soft_prompts=true
```

💡 **最佳效能：** 如果您擁有足夠的運算資源並希望達到最佳的 X-VLA 微調（fine-tune）效能，應遵循官方微調策略（policy）：

**🔥 使用自訂學習率方案全參數微調所有組件（full-finetune all components with a custom learning-rate scheme）**

為確保穩定最佳化（stable optimization），視覺語言模型（VLM）必須使用僅為基礎學習率（base learning rate） 1/10 的學習率（LR）進行訓練（training），而所有其他組件（components）使用完整的學習率（full LR）。
此學習率比例（LR ratio）對於達成強大且穩定的微調效能（finetuning performance）至關重要。預設情況下已為您自動設定好。
❕注意

要完全匹配官方報告的效能，可能需要為軟提示（soft-prompts）額外設置預熱學習率（learning rate）排程（warm-up LR schedule），這能帶來微小的改善。
我們鼓勵在您自訂的訓練管線（customized training pipeline）中實現此功能以獲得最佳結果。

## 核心概念（core concepts）

### 1. 動作模式（action modes）

X-VLA 使用**動作註冊表（Action Registry）**系統來處理不同的動作空間（action spaces）和具身形態（embodiments）。`action_mode` 參數（parameter）定義了動作（actions）的處理方式、使用的損失函數（loss functions），以及預測（predictions）的後處理（post-processed）方式。

#### 可用的動作模式（available action modes）

| 動作模式（action mode） | 動作維度（action dim）              | 描述（description）                                                    | 使用場景（use case）                                     |
| ----------------------- | ----------------------------------- | ---------------------------------------------------------------------- | -------------------------------------------------------- |
| `ee6d`                  | 20                                  | 末端執行器（end-effector）含 xyz、6D 旋轉（rotation）、夾爪（gripper） | 雙臂空間控制設置（dual-arm setups with spatial control） |
| `joint`                 | 14                                  | 關節空間含夾爪（joint-space with gripper）                             | 直接關節控制的機器人（direct joint control robots）      |
| `agibot_ee6d`           | 20                                  | AGI-bot 變體（variant ），使用 MSE 損失（loss）                        | AGI-bot 平台（platforms）                                |
| `so101_bimanual`        | 20（模型 model）、12（實際 real）   | SO101 雙臂機器人（bimanual robot）                                     | 雙臂操控任務（bimanual manipulation tasks）              |
| `auto`                  | 20（模型 model）、自動（實際 real） | 自動從資料集偵測動作維度（auto-detects action dim from dataset）       | **推薦**用於新機器人                                     |

#### 動作模式的重要性（why action modes matter）

當您擁有一個以 `action_dim=20` 訓練的預訓練檢查點（pretrained checkpoint，如 `lerobot/xvla-base`），並希望在具有不同動作維度（action dimension，例如雙臂 bimanual arms 的 14 維）的資料集（dataset）上訓練時，您不能簡單地裁剪動作維度（trim the action dimension）。動作模式（action mode）負責協調：

1. **損失計算（loss computation）**：針對不同動作組件（action components，關節 joint 用 MSE、夾爪 gripper 用 BCE 等）使用不同的損失函數（loss functions）
2. **前處理（preprocessing）**：將夾爪通道（gripper channels）歸零、填充維度（padding dimensions）
3. **後處理（postprocessing）**：對夾爪（gripper） logits 應用 sigmoid、裁剪填充（trimming padding）

#### 範例：BimanualSO101 動作空間（BimanualSO101 action space）

`so101_bimanual` 動作模式（action mode）處理了模型輸出（model output, 20D）與真實機器人控制（real robot control, 12D）之間的不匹配（mismatch）：

```python
# 模型輸出 20 個維度以保持相容性
dim_action = 20

# 真實機器人只需要 12 個維度
# [左臂 (6), 右臂 (6)] = [關節 (5) + 夾爪 (1)] × 2
REAL_DIM = 12

# 前處理：將 12D 動作填充至 20D 用於訓練
# 後處理：將 20D 預測裁剪至 12D 用於部署
```

詳情請參閱 [action_hub.py](https://github.com/huggingface/lerobot/blob/main/src/lerobot/policies/xvla/action_hub.py) 的實作。

#### 推薦：Auto 的動作模式（auto action mode）

`auto` 動作模式（action mode）是在任何機器人上使用 X-VLA 最簡單的方式。它會自動偵測您資料集的動作維度（dataset’s action dimension）並處理填充/裁剪（padding/trimming）：

```bash
lerobot-train \
  --policy.path="lerobot/xvla-base" \
  --policy.action_mode=auto \
  --policy.max_action_dim=20 \
  ...
```

**運作方式（how it works）：**

- 從您的資料集（dataset）讀取 `action_feature.shape[-1]`（例如 Franka 為 7）
- 模型輸出（model outputs） `max_action_dim`（預設 20）以保持預訓練相容性（pretrained compatibility）
- 損失（loss）**僅在實際維度上計算**：`MSE(pred[:,:,:real_dim], target[:,:,:real_dim])`
- 後處理（postprocess）將輸出裁剪（trims）回 `real_dim` 用於機器人控制（robot control）

這使得大多數機器人無需建立自訂動作模式（custom action modes）。

### 2. 領域 ID（domain IDs）

領域 ID 是針對不同機器人配置（robot configurations）和攝影機設置（camera setups）的可學習識別符（learnable identifiers）。它們使 X-VLA 能夠區分：

- 不同的機器人（機器人 1 vs 機器人 2）
- 不同的攝影機配置（cam1 vs cam2）
- 不同的組合（Robot1-cam1-cam2 vs Robot1-cam1 vs Robot2-cam1）

#### 設置領域 ID（setting domain IDs）

**訓練過程中（during training）**：預設情況下，domain_id 設為 0 用於一般訓練（training）。

**評估過程中（during evaluation）**：指定與您的檢查點訓練配置（checkpoint’s training configuration）相匹配的 domain_id。

```python
# 範例：LIBERO 檢查點使用 domain_id=3
domain_id = 3
```

domain_id 會由前處理管線（preprocessing pipeline）中的 `XVLAAddDomainIdProcessorStep` 自動加入觀測（observations）中。

`lerobot/xvla-base` 模型（model）已在以下領域 ID 上進行訓練（training）。建議選擇一個與您的機器人/配置（robot/configuration）最相似的：

#### 微調資料集（fine-tuning datasets）

| 資料集名稱（dataset name） | 領域 ID（domain ID） |

| Bridge                     | 0                    |
| RT1                        | 1                    |
| Calvin                     | 2                    |
| libero                     | 3                    |

### 3. 處理器的步驟（processor steps）

X-VLA 需要特定的前處理和後處理步驟（preprocessing and postprocessing steps）才能正常運作。

#### 必要的前處理步驟（required preprocessing steps）

1. **XVLAImageToFloatProcessorStep**：將影像（images）從 [0, 255] 轉換為 [0, 1] 範圍（range）
2. **XVLAImageNetNormalizeProcessorStep**：應用 ImageNet 正規化（normalization，VLM 主幹網路 backbone 所需）
3. **XVLAAddDomainIdProcessorStep**：將 domain_id 加入觀測（observations）中

#### 自訂處理器範例（example custom processor）

對於 LIBERO 環境（environments），自訂處理器（custom processor）處理特定的觀測格式（observation format）：

```python
from lerobot.policies.xvla.processor_xvla import LiberoProcessorStep

processor = LiberoProcessorStep()
# 處理 robot_state 字典，將旋轉矩陣轉換為 6D 表示
# 應用 180° 影像旋轉以符合相機慣例
```

### 4. 配置參數（configuration parameters）

X-VLA 的關鍵配置參數（configuration parameters）：

```python
# 觀測與動作
n_obs_steps: int = 1          # 觀測時間步數
chunk_size: int = 32           # 動作序列長度
n_action_steps: int = 32       # 要執行的動作步數

# 模型架構
hidden_size: int = 1024        # Transformer 隱藏層維度
depth: int = 24                # Transformer 層數
num_heads: int = 16            # 注意力頭數
num_domains: int = 30          # 最大領域 ID 數量
len_soft_prompts: int = 32     # 軟提示嵌入長度

# 動作空間
action_mode: str = "ee6d"      # 動作空間類型（使用 "auto" 自動偵測）
use_proprio: bool = True       # 使用本體感覺狀態
max_state_dim: int = 32        # 最大狀態維度
max_action_dim: int = 20       # 填充用的最大動作維度（"auto" 模式使用）

# 視覺
num_image_views: int | None    # 相機視角數量
resize_imgs_with_padding: tuple[int, int] | None  # 帶填充的目標影像尺寸

# 訓練
num_denoising_steps: int = 10  # 流匹配去噪步數
```

## 建立自訂動作模式（creating custom action modes）

如果您的機器人擁有獨特的動作空間（unique action space），您可以建立自訂動作模式（custom action mode）：

### 步驟 1：定義您的動作空間（define your action space）

```python
from lerobot.policies.xvla.action_hub import BaseActionSpace, register_action
import torch.nn as nn

@register_action("my_custom_robot")
class MyCustomActionSpace(BaseActionSpace):
    """我的機器人的自訂動作空間。"""

    dim_action = 15  # 您的機器人的動作維度
    gripper_idx = (7, 14)  # 夾爪通道索引

    def __init__(self):
        super().__init__()
        self.mse = nn.MSELoss()
        self.bce = nn.BCEWithLogitsLoss()

    def compute_loss(self, pred, target):
        """定義您的損失計算方式。"""
        # 範例：關節用 MSE，夾爪用 BCE
        joints_loss = self.mse(pred[:, :, :7], target[:, :, :7])
        gripper_loss = self.bce(pred[:, :, self.gripper_idx],
                                target[:, :, self.gripper_idx])

        return {
            "joints_loss": joints_loss,
            "gripper_loss": gripper_loss,
        }

    def preprocess(self, proprio, action, mode="train"):
        """訓練前的動作前處理。"""
        # 範例：將本體感覺中的夾爪歸零
        proprio_m = proprio.clone()
        action_m = action.clone() if action is not None else None
        proprio_m[..., self.gripper_idx] = 0.0
        if action_m is not None:
            action_m[..., self.gripper_idx] = 0.0
        return proprio_m, action_m

    def postprocess(self, action):
        """部署時的預測後處理。"""
        # 範例：對夾爪 logits 應用 sigmoid
        action[..., self.gripper_idx] = torch.sigmoid(action[..., self.gripper_idx])
        return action
```

## 疑難排解

### 常見問題

**問題**：「動作維度不匹配（Action dimension mismatch）」

- **解決方案**：確認您的 `action_mode` 與機器人的動作空間（robot’s action space）相匹配。如有需要，建立自訂動作模式（custom action mode）。

**問題**：「影像數值超出 [0, 1] 範圍（Image values outside [0, 1] range）」

- **解決方案**：確保影像（images）在正規化（normalization）之前已經過 `XVLAImageToFloatProcessorStep` 前處理（preprocesse）。

**問題**：「找不到領域 ID（Domain ID not found）」

- **解決方案**：確認 `XVLAAddDomainIdProcessorStep` 在您的前處理管線（preprocessing pipeline）中，並設定了正確的 domain_id。

**問題**：「新具身形態上的成功率低（Low success rate on new embodiment）」

- **解決方案**：
  1. 確認您的 action_mode 是否正確
  2. 確認軟提示（soft prompts）是否正在訓練（`train_soft_prompts=True`）
  3. 確保正確的前處理（preprocessing，ImageNet 正規化、domain_id）
  4. 考慮增加訓練步數（training steps）

**問題**：「訓練時記憶體不足（Out of memory during training）」

- **解決方案**：
  1. 減少 `chunk_size`（例如從 32 減至 16）
  2. 啟用梯度檢查點（gradient checkpointing）
  3. 減少批次大小（batch size）
  4. 凍結（freeze）更多組件（components）
