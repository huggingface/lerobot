# 重新命名映射表與空白攝影機（rename map and empty cameras）

當您使用機器人策略（robot policy）進行訓練（train）、評估（evaluate）或錄製（record）時，您的**資料集（dataset）**或**環境（environment）**提供的觀測資料（observations）使用一組鍵名（set of keys，例如 `observation.images.front`、`observation.images.eagle`），而您的**策略（policy）**則期望另一組鍵名（例如 `observation.images.image`、`observation.images.image2`）。**重新命名映射表（rename map）**可以在不修改策略或資料來源的情況下橋接這個差異。

> **適用範圍（scope）**：重新命名對應表（rename map）僅重新命名**觀測（observation）**鍵（keys ，影像 image 與狀態 state）。動作鍵（action keys）不受影響。

## 為什麼觀測鍵名不一定匹配（why observation keys don’t always match）

策略（policy）有一組固定的**輸入特徵名稱（input feature names）**，內建於其預訓練設定（pretrained config）中。例如：

- [pi0fast-libero](https://huggingface.co/lerobot/pi0fast-libero) 需要 `observation.images.base_0_rgb` 與 `observation.images.left_wrist_0_rgb`。
- [xvla-base](https://huggingface.co/lerobot/xvla-base) 需要 `observation.images.image`、`observation.images.image2` 以及 `observation.images.image3`。

您的資料集（dataset）可能使用完全不同的名稱（例如 `observation.images.front`、`observation.images.eagle`、`observation.images.glove`），而您的評估環境（eval environment）可能又使用另一組名稱。與其編輯策略設定（policy config）或重新命名資料集中的欄位（renaming columns in the dataset），您可以傳入一個**重新命名對應表（rename map）**：一個 JSON 字典（dictionary），將來源鍵映射到策略期望的鍵（maps source keys to the keys the policy expects）。重新命名在預處理管線（preprocessor pipeline）中進行，因此策略（policy）始終看到它所期望的鍵名（expected keys）。

## 使用重新命名映射表（using the rename map）

在命令列中以 JSON 字串（string）傳入映射（mapping）。慣例格式如下：

```
--rename_map='{"source_key": "policy_key", ...}'
```

其中 **source_key** 是資料集（dataset）或環境（environment）提供的鍵名，而 **policy_key** 是策略（policy）期望的鍵名。

只有列出的鍵會被重新命名；其餘所有鍵不變地直接傳遞。項目的順序（order of entries）無關緊要。

支援的策略（supported policies）：**PI0**、**PI05**、**PI0Fast**、**SmolVLA** 和 **XVLA**。

### 訓練（training）

假設您在一個資料集（dataset）上微調（fine-tune） [lerobot/xvla-base](https://huggingface.co/lerobot/xvla-base)，該資料集的影像（images）在 `observation.images.front`、`observation.images.eagle` 和 `observation.images.glove` 下。XVLA 期望 `observation.images.image`、`observation.images.image2` 和 `observation.images.image3`：

```bash
lerobot-train \
  --dataset.repo_id=YOUR_DATASET \
  --output_dir=./outputs/xvla_training \
  --job_name=xvla_training \
  --policy.path="lerobot/xvla-base" \
  --policy.repo_id="HF_USER/xvla-your-robot" \
  --policy.dtype=bfloat16 \
  --policy.action_mode=auto \
  --steps=20000 \
  --policy.device=cuda \
  --policy.freeze_vision_encoder=false \
  --policy.freeze_language_encoder=false \
  --policy.train_policy_transformer=true \
  --policy.train_soft_prompts=true \
  --rename_map='{"observation.images.front": "observation.images.image", "observation.images.eagle": "observation.images.image2", "observation.images.glove": "observation.images.image3"}'
```

### 評估（evaluation）

一個策略（policy）期望 `observation.images.base_0_rgb` 和 `observation.images.left_wrist_0_rgb`（例如 [pi0fast-libero](https://huggingface.co/lerobot/pi0fast-libero)），但 LIBERO 環境（environment）回傳的是 `observation.images.image` 和 `observation.images.image2`：

```bash
lerobot-eval \
  --policy.path=lerobot/pi0fast-libero \
  --env.type=libero \
  ... \
  --rename_map='{"observation.images.image": "observation.images.base_0_rgb", "observation.images.image2": "observation.images.left_wrist_0_rgb"}'
```

## 替代方案：直接編輯策略配置（edit the policy config directly）

如果您總是使用相同的資料集（dataset）或環境（environment），您可以**編輯策略的 `config.json`**，使其觀測鍵名（observation keys）與您的資料來源匹配。這樣就不需要重新命名映射表（rename map）。

取捨在於（tradeoff）：修改策略設定（policy config）會將其綁定到單一資料來源。重新命名對應表（rename map）則讓一個策略可以跨多個資料集（dataset）和環境（environment）使用。

## 空白攝影機：視角數少於策略期望的數量（empty cameras: fewer views than the policy expects）

有些策略（policy）針對固定數量的影像輸入（image inputs）而建構。如果您的資料集（dataset）擁有較少的攝影機，您可以在策略設定中設定 **`empty_cameras`**，而不需要修改模型架構（model architecture）。

### 運作原理（how it works）

設定 `empty_cameras=N` 會在策略設定（policy config）中新增 N 個佔位影像特徵（placeholder image features），命名為：

```
observation.images.empty_camera_0
observation.images.empty_camera_1
...
```

在執行時，這些鍵在批次（batch）中沒有對應的資料。策略（policy）會以遮罩的虛擬張量（masked dummy tensors）填充它們（對於基於 SigLIP 的視覺編碼器 vision encoders，以 `-1` 填充，注意力遮罩 attention mask 設為零），因此額外的影像插槽（image slots）在訓練（training）和推論（inference）期間實際上會被忽略。

### 範例（example）

XVLA-base 有三個視覺輸入（visual inputs），預設 `empty_cameras=0`。您的資料集（dataset）只有兩個攝影機：

1. 設定 `--policy.empty_cameras=1`。
2. 配置（config）會新增第三個鍵：`observation.images.empty_camera_0`。
3. 照常使用重新命名映射表（rename map）對應您的兩個實際攝影機（real cameras）。
4. 第三個插槽（slot）會被遮罩掉（masked out ）——您的資料集（dataset）中不需要假影像（fake images）。

## 快速參考（quick reference）

| 目標（goal）                                         | 怎麼做（what to do）                                                 |
| ---------------------------------------------------- | -------------------------------------------------------------------- |
| 資料集鍵名（dataset keys） ≠ 策略鍵名（policy keys） | `--rename_map='{"dataset_key": "policy_key", ...}'`                  |
| 環境鍵名（env keys） ≠ 策略鍵名（policy keys, eval） | `--rename_map='{"env_key": "policy_key", ...}'`                      |
| 推演（rollout）時使用不同鍵名（inference）           | `--rename_map='{"source_key": "policy_key", ...}'`                   |
| 攝影機數量少於策略期望（policy expects）             | `--policy.empty_cameras=N`（PI0、PI05、PI0Fast、SmolVLA、XVLA 支援） |
| 避免傳入重新命名對應表（rename map）                 | 編輯策略的 `config.json`，使其鍵名（keys）與您的資料來源匹配         |
