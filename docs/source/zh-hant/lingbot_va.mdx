# LingBot-VA

LingBot-VA 是一個建構於 **Wan2.2** 影片擴散堆疊（video-diffusion stack）之上的
**自迴歸影片-動作世界模型策略（autoregressive video-action world-model policy）**。
它在單一自迴歸序列（autoregressive sequence）中，交錯（interleaves）預測未來的
**影片潛在表徵（video latent）**與**機器人動作（robot actions）**（「VA」＝ Video-Action）。
此 LeRobot 整合將 LingBot-VA 接入標準的訓練（training）、評估（evaluation）與處理器介面（processor interfaces）。

## 模型概覽（model overview）

LingBot-VA 是一個**雙串流 transformer 混合體（dual-stream “mixture-of-transformers”）**：一個影片／潛在表徵串流
（video/latent stream, `patch_embedding_mlp → blocks → proj_out`）與一個動作串流
（action stream, `action_embedder → blocks → action_proj_out`）共用相同的 30 個 transformer 區塊（blocks）與
文字條件（text conditioning）。

| 元件（component）                       | 類（class）             | 角色（role）                                                            |
| --------------------------------------- | ----------------------- | ----------------------------------------------------------------------- |
| DiT 骨幹（backbone，可訓練 trainable）  | `WanTransformer3DModel` | 約 50 億參數的雙串流（dual-stream） transformer。                       |
| VAE（凍結 frozen）                      | `AutoencoderKLWan`      | Wan2.2 VAE，`z_dim=48`。於載入時從來源儲存庫延遲拉取（lazy-pulled）。   |
| 文字編碼器（text encoder，凍結 frozen） | `UMT5EncoderModel`      | UMT5-XXL，`d_model=4096`。於載入時從來源儲存庫延遲拉取（lazy-pulled）。 |

在推論（inference）時，此策略（policy）以每個塊為單位執行自迴歸迴圈（runs an autoregressive loop per chunk）：它以兩個獨立的流匹配排程器
（flow matching scheduler），對影片潛在表徵串流（video-latent stream，CFG，約 20 步）與動作串流（action stream，約 50 步）去雜訊（denoise），
並跨塊維持一個 KV 快取（KV cache）。在執行塊（chunk）的過程中，真實觀測到的關鍵影格（Real observed keyframes）
會被回饋到 KV 快取中（閉迴路世界建模，closed-loop world modeling）。

### LeRobot 整合涵蓋的內容（what the LeRobot integration covers）

- 透過 LeRobot 的標準 `policy.type=lingbot_va` 設定（configuration）。
- Hub 上即用的 LeRobot 格式檢查點（checkpoints，由已發布的上游 released upstream 版本轉換而來）。
- 在標準 `select_action` 介面（interface）背後的自迴歸雙串流推論（autoregressive dual-stream inference；
  單一環境評估 single-environment eval，`--eval.batch_size=1`）。
- 在評估／訓練（eval / training）期間可選擇性地儲存策略的**預測（想像）影片（predicted, imagined videos）**。
- 以 `lerobot-eval` 在 LIBERO 與 RoboTwin 上進行評估（evaluation）。
- 透過雙串流流匹配損失（dual-stream flow-matching loss, `policy.forward`）進行訓練／微調（training / fine-tuning），詳見下文。

## 安裝（installation）

1. 按照[安裝指南（Installation Guide）](./installation)安裝 LeRobot。
2. 安裝 LingBot-VA 額外套件（extra）：

```bash
pip install -e ".[lingbot_va]"
```

## 檢查點（checkpoints）

已發布的上游檢查點（released upstream checkpoints）已轉換為 LeRobot 格式（format）並推送到 Hub：

| 變體（variant）                  | LeRobot 檢查點（checkpoint）     |
| -------------------------------- | -------------------------------- |
| LIBERO-Long 後訓練（post-train） | `lerobot/lingbot_va_libero_long` |
| RoboTwin 後訓練（post-train）    | `lerobot/lingbot_va_robotwin`    |
| 預訓練基礎版（pretrained base）  | `lerobot/lingbot_va_base`        |

LeRobot 的 `model.safetensors` 只儲存可訓練的約 50 億（5B）參數 transformer。凍結（frozen）的 VAE + UMT5 +
tokenizer（約 20 GB）會在載入時從 `config.wan_pretrained_path` 拉取
（預設為來源 `robbyant/*` 儲存庫 repo）。UMT5-XXL 文字編碼器（text encoder）預設在 CPU 上執行
（`config.text_encoder_device`），因此 50 億（5B）參數 transformer + VAE 可容納於單張
24–32 GB GPU。

## 評估：LIBERO（evaluation on LIBERO）

```bash
lerobot-eval \
    --policy.path=lerobot/lingbot_va_libero_long \
    --policy.device=cuda \
    --env.type=libero --env.task=libero_10 \
    --env.observation_height=128 --env.observation_width=128 \
    --eval.n_episodes=50 --eval.batch_size=1 \
    --output_dir=outputs/eval/lingbot_va_libero
```

LingBot-VA 的串流推論（KV 快取 + 觀測關鍵影格回饋）是為單一環境評估而實作的；
請使用 `--eval.batch_size=1`。

## 評估：RoboTwin（evaluation on RoboTwin）

RoboTwin 2.0 需要 SAPIEN + CuRobo 模擬器堆疊（simulator stack）。您可以使用基準測試 Docker 映像檔
（benchmark Docker image, `docker/Dockerfile.benchmark.robotwin`，它還需要 `warp-lang==1.3.1`，
以及在 `TORCH_CUDA_ARCH_LIST` 中以 GPU 運算能力建置的 CuRobo）。RoboTwin 使用**末端執行器姿態控制（end-effector-pose control）**，
因此請以 `--env.action_mode=ee` 執行：策略（policy）會預測每隻手臂的 `xyz+quaternion+gripper` 增量
（deltas；`robotwin_tshape` 潛在表徵佈局 latent layout），這些增量會疊加到該回合的初始 eef 姿態（episode’s initial eef pose）上，
並透過 CuRobo IK 執行。

```bash
lerobot-eval \
    --policy.path=lerobot/lingbot_va_robotwin \
    --policy.device=cuda \
    --env.type=robotwin --env.task=beat_block_hammer --env.action_mode=ee \
    --eval.n_episodes=10 --eval.batch_size=1 \
    --output_dir=outputs/eval/lingbot_va_robotwin
```

### 儲存預測（想像）影片（saving predicted, imagined videos）

設定 `--policy.save_predicted_video=true` 可額外將預測的影片潛在表徵（latent）以 VAE 解碼，
並在環境渲染的 `eval_episode_*.mp4` 影片旁寫入 `pred_episode_*.mp4`。
同一個旗標（flag）也適用於 `lerobot-train` 期間的週期性評估。

## 訓練／微調（training / fine-tuning）

`LingBotVAPolicy.forward(batch)` 實作了論文中的雙串流**流匹配**損失（dual-stream flow-matching loss,
`latent_loss + action_loss`，以時間步加權 timestep-weighted、動作遮罩 action-masked）：它將攝影機片段（camera clips）
以 VAE 編碼（VAE-encodes）為影片潛在表徵（video latents），將任務以 UMT5 編碼（UMT5-encodes），
對兩個串流（streams）加入雜訊（noises），執行 transformer 的區塊因果訓練通道（transformer’s block-causal training pass），並回傳 `(loss, metrics)`。
優化器（optimizer）預設為 AdamW，搭配線性暖身後恆定的排程（linear-warmup-then-constant schedule，與上游一致）。

需求（requirements）：

- 區塊因果遮罩（block-causal masks）使用 PyTorch 的 **flex-attention**，因此訓練（training）時請以
  `--policy.attn_mode=flex` 建構策略（build the policy；預設的 `torch` SDPA 僅供推論 inference-only）。
- 完整的 50 億（5B）參數 DiT 在 AdamW 下無法容納於單張 24–32 GB GPU；請以 **LoRA**（`--policy.use_peft=true`）
  和／或優化器卸載（optimizer offload）進行微調。`get_optim_params` 只回傳可訓練（trainable；例如轉接器 adapter）的參數（parameters）；
  VAE + UMT5 文字編碼器維持凍結（text encoder stay frozen）。

```bash
lerobot-train \
  --policy.path=lerobot/lingbot_va_libero_long --policy.attn_mode=flex \
  --policy.use_peft=true \
  --dataset.repo_id=<your LeRobot-format dataset> \
  --batch_size=1 --steps=... --output_dir=outputs/train/lingbot_va
```

資料集（dataset）必須提供攝影機片段（camera clips，每個攝影機一個時間視窗 temporal window，經 VAE 編碼為 `frame_chunk_size` 個潛在影格 latent frames），
以及每個項目（item） `frame_chunk_size * action_per_frame` 個動作步（action steps）。

## 資料格式（動作通道 & 攝影機順序，action channels & camera order）

LingBot-VA 是一個**末端執行器（笛卡兒）姿態（end-effector Cartesian pose）**策略，它預測 EEF 姿態（EEF poses） + 夾爪（gripper），而非關節位置（joint positions）。
動作採用固定的多具身（multi-embodiment）**30 維（30-dim）**佈局（layout）；請將您機器人的動作維度（action dimensions）
對映到這些通道（channels），其餘以 `0` 填補（`used_action_channel_ids` 會選出某個檢查點 checkpoint
實際使用的通道 channels）：

| 通道（channels） | 意義（meaning）                                                          |
| ---------------- | ------------------------------------------------------------------------ |
| 0–6              | 左臂末端執行器姿態（left-arm end-effector pose）                         |
| 7–13             | 右臂末端執行器姿態（right-arm end-effector pose）                        |
| 14–20            | 左臂關節（left-arm joints，已發布的檢查點 released checkpoints 未使用）  |
| 21–27            | 右臂關節（right-arm joints，已發布的檢查點 released checkpoints 未使用） |
| 28               | 左夾爪（left gripper）                                                   |
| 29               | 右夾爪（right gripper）                                                  |

- **LIBERO** 使用通道 `0–6`：6 自由度的 EEF 增量（6-DoF EEF delta，xyz + 旋轉 rotation）+ 夾爪（gripper，單臂 single arm）。
- **RoboTwin** 使用通道 `[0–6, 28, 7–13, 29]`：左 EEF（xyz + 四元數 quaternion）+ 左夾爪（gripper） + 右 EEF +
  右夾爪（gripper）共16 維（dims）。環境（env ）會透過 CuRobo IK 將這些姿態（poses）轉換為關節軌跡（joint trajectories），
  從不預測關節（joints）。

關節空間的資料集（joint-space dataset，或不同的 EEF 慣例 convention）在微調（fine-tuning）這些檢查點（checkpoints）之前，
必須重新對映（remapped ）到此 schema。

**攝影機順序是固定且對順序敏感的（camera order is fixed and order-sensitive）**，各攝影機的潛在表徵（per-camera latents）會依 `obs_cam_keys` 的順序（order）在空間上串接（concatenated spatially），
因此實體攝影機（physical camera）→插槽的對映（slot mapping）必須與訓練（training）時相符：

| 基準測試 | `obs_cam_keys`（依序）                                                                                | `camera_layout`                                                                               |
| -------- | ----------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------- |
| LIBERO   | `observation.images.image`（agentview／第三人稱）、`observation.images.image2`（手上 eye-in-hand）    | `width_concat`（潛在表徵 latents 沿寬度 width 串接 concatenate）                              |
| RoboTwin | `observation.images.head_camera`、`observation.images.left_camera`、`observation.images.right_camera` | `robotwin_tshape`（下方為全解析度頭部 full-res head，上方為兩個半解析度手腕 half-res wrists） |

第一個攝影機是外部／頭部視角（exterior/head view），其餘為手腕視角（wrist views）。

## 推論超參數（LIBERO）（inference hyperparameters on LIBERO）

| 鍵（key）                              | 值（value）                                                                         |
| -------------------------------------- | ----------------------------------------------------------------------------------- |
| height × width                         | 128 × 128                                                                           |
| cameras                                | `observation.images.image`（agentview）、`observation.images.image2`（eye-in-hand） |
| action channels used                   | 0–6（7-DoF 手臂 arm + 夾爪 gripper）                                                |
| action_per_frame / frame_chunk_size    | 4 / 4                                                                               |
| attn_window                            | 30                                                                                  |
| video / action denoising steps         | 20 / 50                                                                             |
| guidance_scale / action_guidance_scale | 5 / 1                                                                               |
| snr_shift / action_snr_shift           | 5.0 / 0.05                                                                          |

這些是 `LingBotVAConfig` 的預設值；可透過 `--policy.<name>=...` 覆寫其中任何一項。

## 注意事項（notes）

- **注意力後端（attention backend）：** 推論（inference）使用 `torch` SDPA 後端（backend；一律可用）。
  `flashattn` 與 `flex` 後端為可選；`flex` 只有訓練（training）時需要。
- **模型大小（model size）：** DiT 約有 50 億（5B）參數，凍結的（frozen） VAE+UMT5 再增加約 20 GB；
  推論（inference）大約需要 18–24 GB 的 VRAM。
